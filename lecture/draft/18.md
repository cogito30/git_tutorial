$$\[1편\] 왜 Git으로 포트폴리오를 만드나요? 🚀$$

안녕하세요! 개발자로서 첫 취업 전선에 뛰어들거나 나만의 프로젝트를 대중에게 공개하고 싶을 때, 가장 먼저 필요성을 느끼는 것이 바로 '포트폴리오 사이트'입니다.

보통은 Notion 페이지를 깔끔하게 다듬거나, PDF 파일로 이력서를 첨부하곤 합니다. 물론 이 방법도 좋지만, 기술을 다루는 엔지니어라면 조금 더 강력한 무기가 필요합니다. 바로 '내가 직접 코드를 짜서 Git으로 관리하고 배포한 포트폴리오 웹사이트'입니다.

첫 번째 편에서는 포트폴리오를 왜 굳이 Git으로 관리해야 하는지, 그리고 우리 프로젝트의 튼튼한 첫 단추를 채우는 방법을 알아보겠습니다.

## 1. 채용 담당자가 이력서에서 진짜 보고 싶어 하는 것 🔍
수많은 신입 개발자의 이력서를 검토하는 채용 담당자(리드 개발자)의 입장이 되어봅시다.
- **A 지원자**: "저는 Git을 다룰 줄 압니다. 협업 능력이 뛰어납니다." (이력서 텍스트 기술)
- **B 지원자**: 자신의 포트폴리오 GitHub 저장소 링크를 제출함. 들어가 보니 기능 단위로 예쁘게 쪼개진 브랜치, 스스로 올린 정성스러운 Pull Request(PR) 기록, 그리고 무엇보다 날짜별로 깔끔하게 정리된 커밋 히스토리가 살아있음.

여러분이라면 어떤 지원자에게 더 신뢰가 가고 면접 기회를 주고 싶으신가요? 당연히 **B 지원자**입니다.

포트폴리오를 Git 기반으로 제작하고 관리한다는 것은 단순히 '코드를 저장한다'는 것을 넘어 "나는 실무 프로세스를 이해하고 있고, 내 코드의 이력을 투명하고 정교하게 관리할 줄 아는 준비된 엔지니어다"라는 것을 이력서 한 줄 없이도 100% 증명해 내는 과정입니다.

## 2. 포트폴리오 기획 및 기술 스택 선정 🛠️
우리가 만들 포트폴리오 사이트는 너무 거창할 필요가 없습니다. 하지만 '깔끔함'과 '전달력'이 생명입니다.

#### 📋 사이트에 꼭 들어가야 할 핵심 3요소
1. **소개 (About Me)**: 나를 한 단어로 표현하는 헤드라인과 간단한 자기소개, 기술 블로그 및 SNS 링크
2. **기술 스택 (Tech Stacks)**: 내가 단순히 '단어만 들어본 기술'이 아니라, '실제로 프로젝트에 활용할 수 있는 기술'을 수준별로 솔직하게 정리
3. **프로젝트 목록 (Projects)**: 핵심 프로젝트들의 썸네일, 주요 역할, 사용 기술, 그리고 상세 회고(Troubleshooting) 링크

#### 🎨 기술 스택 고르기
이번 튜토리얼은 **누구나 쉽게 따라 할 수 있으면서도 실무적인 빌드 도구를 활용하는 방향**으로 진행합니다. 다음 중 여러분의 현재 수준에 맞는 구성을 선택해 보세요!
- **입문자 추천 (HTML5 + Tailwind CSS + Vanilla JS)**: 별도의 빌드 도구 없이, 깔끔하고 트렌디한 CSS 프레임워크인 Tailwind를 활용해 빠르게 반응형 웹사이트를 제작합니다.
- **프론트엔드 지망생 추천 (React / Next.js / Vue)**: 컴포넌트 단위 설계 능력과 빌드 자동화 파이프라인을 보여주고 싶다면 React나 Next.js를 추천합니다.

본 튜토리얼에서는 가장 직관적이면서도 현대적인 웹사이트 구성을 위해 *HTML + Tailwind CSS* 방식을 기본으로 설명하되, React 등 프레임워크 사용자들을 위한 팁도 함께 곁들일 예정입니다.

## 3. 포트폴리오의 시작: GitHub 저장소 생성 및 첫 커밋  🌱
자, 이제 기획은 끝났습니다. 머릿속에 그린 첫 화면을 세상에 선보이기 위해 저장소를 셋업해 봅시다.

#### 1단계: GitHub 원격 저장소 생성
1. 내 GitHub 계정에 로그인한 뒤, **\[New repository\]** 버튼을 클릭합니다.
2. 저장소 이름은 나의 정체성을 나타낼 수 있게 깔끔하게 짓습니다. (예: `my-portfolio` 또는 `username.github.io`)
3. Public 상태로 설정하고, 아래 옵션들(`Add a README.md`, `.gitignore`, `Choose a license`)은 **모두 체크하지 않은 상태**로 둡니다. (로컬에서 직접 세팅하는 프로의 모습을 보여주기 위함입니다!)
4. **\[Create repository\]** 버튼을 눌러 저장소를 생성합니다.

#### 2단계: 로컬 프로젝트 생성 및 첫 커밋 날리기
이제 내 컴퓨터에 작업 폴더를 만들고, 원격 저장소와 연결하여 영광스러운 첫 번째 버전을 기록할 시간입니다. 터미널을 열고 다음 명령어들을 차례대로 입력해 보세요.
```
# 1. 작업할 폴더 만들기 및 이동
mkdir my-portfolio
cd my-portfolio

# 2. 로컬 Git 저장소로 초기화하기
git init

# 3. 프로젝트의 대문이 될 메인 마크다운 파일 생성
echo "# My Portfolio Website" > README.md

# 4. 변경 사항 카트에 담기
git add README.md

# 5. 역사적인 첫 커밋 남기기 (컨벤션 적용!)
git commit -m "chore: 프로젝트 초기화 및 README 작성"

# 6. 기본 브랜치 이름을 main으로 변경
git branch -M main

# 7. GitHub 원격 저장소 연결하기 (origin 주소는 본인의 것으로 입력해 주세요)
git remote add origin https://github.com/여러분의ID/my-portfolio.git

# 8. 대망의 첫 푸시 날리기!
git push -u origin main
```
원격 저장소에 들어가서 새로고침을 해보세요. 방금 작성한 `README.md` 파일과 함께 우리가 정성스레 작성한 첫 커밋 로그가 예쁘게 올라가 있는 모습을 볼 수 있을 것입니다!

## 🎯 오늘 배운 내용 요약
1. 포트폴리오는 단순히 완성본을 보여주는 것을 넘어, **Git 히스토리를 통해 나의 개발 여정과 협업 역량을 증명하는 가장 확실한 무기**이다.
2. 사이트는 **소개, 기술 스택, 프로젝트 회고** 중심의 담백하지만 강력한 구조로 기획한다.
3. 로컬에서 직접 `git init`을 하고 `README.md` 파일을 담아 `chore: 프로젝트 초기화 및 README 작성`이라는 첫 흔적을 세상에 남겼다.

첫 단추가 아주 훌륭하게 채워졌습니다!

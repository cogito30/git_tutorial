[실무 Git 튜토리얼 9편] 저 커밋만 쏙 빼올래: Cherry-pick 🍒

안녕하세요! 지난 8편에서 다뤘던 Rebase 마법은 잘 연습해 보셨나요?

오늘 배울 명령어는 이름부터 아주 달콤한 Cherry-pick(체리픽)입니다. 케이크 위에 올려진 맛있는 체리만 쏙 빼먹는 것처럼, 수많은 커밋 히스토리 중에서 '내가 딱 원하는 커밋 하나'만 골라서 내 브랜치로 가져오는 기능입니다.

과연 실무에서 이 기능이 언제, 왜 필요할지 아주 현실적인 시나리오를 통해 알아보겠습니다!

1. 당황스러운 실무 시나리오 💦

여러분은 내일 당장 서비스 배포(Release)를 앞두고 main 브랜치를 점검하고 있습니다.
그런데 아뿔싸! 치명적인 로그인 버그를 발견했습니다.

당황해서 사수에게 달려갔더니 사수님이 이렇게 말합니다.

👨‍💻 사수: "아, 그 로그인 버그요? 제가 어제 feature/payment (결제 기능) 브랜치에서 작업하다가 발견해서 미리 고쳐서 커밋해 뒀어요! 그거 가져다 쓰세요."

자, 여기서 딜레마가 생깁니다.
feature/payment 브랜치에는 사수님이 고친 '로그인 버그 수정' 커밋도 있지만, 아직 미완성인 '결제 기능' 커밋들도 잔뜩 들어있습니다.

이 상태로 main 브랜치에 git merge feature/payment를 해버리면?
버그는 고쳐지겠지만, 완성되지 않은 결제 기능까지 통째로 배포되어 버리는 대참사가 발생합니다. 😱

우리는 오직 "로그인 버그를 고친 딱 그 커밋 하나"만 main으로 가져와야 합니다. 이때 쓰는 것이 바로 Cherry-pick입니다!

2. 🍒 Cherry-pick 실전 사용법

자, 사수님의 브랜치에서 원하는 커밋만 쏙 빼와서 배포를 무사히 마쳐봅시다.

🔍 Step 1. 가져올 커밋의 '고유 번호(Hash)' 찾기

먼저 사수님이 작업하던 브랜치의 기록을 보고, 내가 훔쳐올(?) 커밋의 번호를 알아내야 합니다.

# 사수님의 브랜치 기록을 확인합니다.
git log feature/payment


commit 7c8d9e0 (HEAD -> feature/payment)
Author: 사수님
Date:   Wed Jul 1 10:00:00 2026

    feat: 결제 모듈 연동 중... (미완성)

commit 3a4b5c6  <-- [우리가 훔쳐올 커밋 번호!]
Author: 사수님
Date:   Tue Jun 30 15:30:00 2026

    fix: 로그인 안되는 치명적 버그 수정

commit 1f2e3d4
Author: 사수님
...


로그인 버그를 고친 커밋의 번호(Hash) 앞자리인 3a4b5c6를 복사하거나 메모해 둡니다.

🎯 Step 2. 내 브랜치로 가져오기 (Cherry-pick)

이제 다시 내가 코드를 추가할 목적지(main)로 돌아와서 체리를 따옵니다.

# 1. 목적지 브랜치로 이동
git switch main

# 2. 체리픽 실행! (git cherry-pick 커밋번호)
git cherry-pick 3a4b5c6


성공했다면 짠! 🎉 main 브랜치에 fix: 로그인 안되는 치명적 버그 수정 이라는 커밋이 똑같이 복제되어 추가됩니다. (미완성 결제 기능 코드는 쏙 빼놓고 버그 수정 코드만 안전하게 가져왔습니다!)

3. 🚨 충돌(Conflict)이 발생했다면?

Merge나 Rebase를 할 때처럼, Cherry-pick을 할 때도 충돌이 날 수 있습니다. 가져온 커밋의 내용이 내 브랜치의 현재 코드와 겹치면 Git이 혼란에 빠지기 때문이죠.

당황하지 말고 4편에서 배운 대로 해결하면 됩니다.

VS Code 등의 에디터에서 빨갛게 충돌 난 파일의 코드를 알맞게 수정하고 저장합니다.

git add . 명령어로 해결된 파일을 카트(Staging Area)에 담습니다.

주의! commit을 새로 하는 게 아닙니다. 아래 명령어로 체리픽을 마저 진행합니다.

git cherry-pick --continue


💡 꿀팁: 만약 충돌이 너무 심해서 "아, 그냥 안 가져올래!" 하고 체리픽을 취소하고 싶다면 git cherry-pick --abort 를 치면 체리픽을 시도하기 전의 원래 상태로 깨끗하게 돌아갑니다.

4. 💡 실무에서 Cherry-pick을 쓰는 또 다른 경우들

커밋할 브랜치를 착각했을 때:
main 브랜치에서 모르고 새 기능을 열심히 커밋해 버렸습니다. 이때 올바른 브랜치(feature)를 새로 만들고, main에서 잘못 짠 커밋을 cherry-pick으로 가져온 뒤 main의 커밋은 지워버리면 됩니다.

다른 사람의 유용한 코드만 빌려올 때:
동료가 작업 중인 브랜치에 아주 유용한 '공통 함수(Utility)' 커밋이 있습니다. 그 기능만 내 브랜치로 쏙 빼와서 개발 속도를 높일 수 있습니다.

🎯 오늘 배운 내용 요약

브랜치를 통째로 합치지 않고 특정 커밋만 쏙 빼올 때는 Cherry-pick을 쓴다.

사용할 커밋의 고유 번호(Hash)를 알아낸 뒤 👉 git cherry-pick 커밋번호

충돌 시 해결 후 👉 git add . 한 뒤 git cherry-pick --continue

이제 여러분은 다른 브랜치에 있는 코드가 탐날 때 언제든지 원하는 부분만 훔쳐올(?) 수 있는 스킬을 얻었습니다!

하지만 실무를 하다 보면 남의 코드를 가져오는 것보다, 내가 방금 친 커밋을 되돌리고 싶어 눈물 흘리는 순간이 훨씬 많습니다.

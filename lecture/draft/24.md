과거의 나를 세탁하기: Interactive Rebase와 Squash 🧹✨

안녕하세요! 지난 6편에서 우리는 GitHub Actions를 활용해 `git push` 한 번으로 전 세계에 내 사이트를 실시간 배포하는 자동화 파이프라인(CI/CD)을 완벽하게 구축했습니다.

이제 우리 포트폴리오 사이트는 기술적으로 흠잡을 곳 없이 훌륭하게 굴러가고 있습니다. 하지만 배포 자동화까지 만들어둔 저장소를 시니어 개발자가 열어보았을 때, 이런 커밋 로그가 찍혀 있다면 어떨까요?
- `feat: 프로젝트 섹션 추가`
- `fix: 오타 수정`
- `fix: 진짜 진짜 오타 수정`
- `fix: 아 css 왜 안먹지`
- `test: 배포 테스트`

개발 과정에서는 당연히 수많은 시행착오를 겪기 마련이고, 그 흔적들이 커밋으로 고스란히 남게 됩니다. 하지만 이 지저분한 '낙서' 같은 기록들을 최종 병합 브랜치인 `main`에 그대로 남겨두는 것은 프로답지 못합니다.

오늘은 과거의 부끄러운 흔적들을 말끔히 지우고, 여러 개의 자잘한 커밋들을 의미 있는 하나의 깔끔한 커밋으로 합치는 **Interactive Rebase**와 **Squash** 기술을 배워보겠습니다.

## 1. Squash가 무엇인가요? "$N \to 1$의 마법" 🪄
Squash(스쿼시)는 영어로 '짓누르다, 으깨다'라는 뜻을 가지고 있습니다. Git에서의 Squash는 여러 개의 커밋을 단 **하나의 커밋으로 으깨어 합치는 기능**을 말합니다.

예를 들어, 내가 포트폴리오의 '방명록 기능'을 만들면서 아래와 같이 5개의 커밋을 남겼다고 해봅시다.

$$\text{Commit A (기능 초안)} \to \text{Commit B (오타 수정)} \to \text{Commit C (디자인 수정)} \to \text{Commit D (버그 해결)} \to \text{Commit E (완성)}$$

동료 개발자나 채용 담당자가 궁금한 것은 "방명록 기능이 어떻게 완성되었는가"이지, 내가 도중에 몇 번 오타를 냈고 디자인을 어떻게 바꿨는지가 아닙니다.

이 5개의 자잘한 커밋들을 Squash하면, 지저분한 중간 과정은 싹 사라지고 다음과 같은 단 하나의 명확한 커밋으로 깔끔하게 정리됩니다.

$$\text{feat: 방명록 작성 및 조회 기능 구현}$$

## 2. Interactive Rebase (git rebase -i) 시작하기 🛠️

여러 개의 커밋을 합치기 위해 우리는 Git의 상급 도구인 Interactive Rebase(대화형 리베이스)를 사용합니다. 단어 뜻 그대로 Git과 대화하듯이 소통하며 과거 히스토리를 내 마음대로 주무르는 기능입니다.

#### 1단계: 내 커밋 상태 확인하기
먼저 정리하고 싶은 커밋들이 얼마나 쌓여 있는지 `git log`를 한 줄로 예쁘게 확인해 봅니다.
```
git log --oneline -n 5
```
(결과 예시)
```
a1b2c3d (HEAD -> feature/guestbook) fix: 방명록 스크롤 버그 수정
e4f5g6h style: 방명록 입력 창 패딩 조절
j7k8l9m fix: 오타 수정
n0o1p2q feat: 방명록 마크업 및 기본 UI 추가
z9y8x7w (main) docs: README 업데이트
```

우리의 목표는 `main` 브랜치 이후에 쌓인 위 4개의 자잘한 커밋(`n0o1p2q`부터 `a1b2c3d`까지)을 단 하나의 멋진 커밋으로 합치는 것입니다.

#### 2단계: 대화형 리베이스 명령서 발송
최근 4개의 커밋을 조작하기 위해 아래 명령어를 입력합니다.
```
# HEAD(현재 위치)로부터 4개 이전까지의 커밋들을 조작하겠다는 의미입니다.
git rebase -i HEAD~4
```

## 3. 📝 명령서 작성하기 (Squash 적용)
위 명령어를 치면 터미널 창 내부에 낯선 텍스트 에디터(보통 `Vim` 또는 내가 설정한 VS Code 에디터)가 열리며 다음과 같은 내용이 나타납니다.

🚨 **주의**
- 대화형 리베이스 창에서는 커밋 순서가 git log와 반대로, **오래된 커밋이 맨 위**에 있고 **가장 최근 커밋이 맨 아래**에 위치합니다.
```
pick n0o1p2q feat: 방명록 마크업 및 기본 UI 추가
pick j7k8l9m fix: 오타 수정
pick e4f5g6h style: 방명록 입력 창 패딩 조절
pick a1b2c3d fix: 방명록 스크롤 버그 수정

# Commands:
# p, pick <commit> = use commit
# r, reword <commit> = use commit, but edit the commit message
# e, edit <commit> = use commit, but stop for amending
# s, squash <commit> = use commit, but meld into previous commit
```

여기서 각 커밋 맨 앞의 `pick`이라는 글자가 명령어입니다. `pick`은 "이 커밋을 그대로 살려두겠다"는 의미입니다.

우리는 가장 처음 만든 뼈대 커밋(`n0o1p2q`)만 남기고, 그 아래의 자잘한 커밋들을 첫 커밋 안으로 합쳐서 짓누를 것입니다. 따라서 두 번째 줄부터의 `pick`을 `squash`(또는 단축어 `s`)로 변경해 줍니다.
```
pick n0o1p2q feat: 방명록 마크업 및 기본 UI 추가
squash j7k8l9m fix: 오타 수정
squash e4f5g6h style: 방명록 입력 창 패딩 조절
squash a1b2c3d fix: 방명록 스크롤 버그 수정
```

이렇게 수정했다면 파일을 저장하고 에디터를 닫습니다. (Vim 편집기 기준: `Esc` 누른 후 `:wq` 입력 후 엔터)

## 4. ✍️ 새롭고 우아한 커밋 메시지 작성하기
저장하고 닫으면 곧바로 두 번째 에디터 창이 열립니다. 합쳐진 새로운 통합 커밋의 이름을 무엇으로 지을지 결정하는 단계입니다.

에디터 안에는 우리가 합쳤던 원래 커밋들의 메시지가 모두 모여있을 것입니다.
```
# This is a combination of 4 commits.
# This is the 1st commit message:

feat: 방명록 마크업 및 기본 UI 추가

# This is the commit message #2:

fix: 오타 수정

# This is the commit message #3:

style: 방명록 입력 창 패딩 조절

# This is the commit message #4:

fix: 방명록 스크롤 버그 수정
```

여기에 적힌 지저분한 메시지들을 과감히 다 지우고, 우리가 2편에서 약속했던 **커밋 컨벤션에 맞춘 단 하나의 우아한 메시지**로 작성해 줍니다.
```
feat: 방명록 방명 및 조회 기능 구현

- 방명록 UI 레이아웃 마크업 추가 및 반응형 적용
- 모바일 화면에서의 인풋 스크롤 버그 수정 및 입력 가독성 개선
```

메시지 작성을 마친 뒤 저장하고 에디터를 닫으면 터미널에 `Successfully rebased and updated...` 라는 영롱한 메시지가 뜹니다!

#### 🔍 결과 확인하기

다시 한 번 `git log --oneline`을 쳐보세요.
```
# 이전의 지저분한 4개 커밋이 사라지고 완벽한 1개의 커밋만 남았습니다!
b5c6d7e (HEAD -> feature/guestbook) feat: 방명록 방명 및 조회 기능 구현
z9y8x7w (main) docs: README 업데이트
```

#### 🚨 \[경고\] Rebase의 절대 황금률 (Golden Rule)

Interactive Rebase는 우리의 더러운 흔적을 완벽하게 지워주는 강력한 기술이지만, 그만큼 위험성도 큽니다. 실무에서 이 규칙을 어기면 협업하는 동료들의 Git 전체를 꼬이게 만들어 "민폐 개발자"로 찍힐 수 있습니다.
- 🚫 **"이미 GitHub(원격 저장소)에 Push하여 남들과 공유한 커밋은 절대로 Rebase 하지 마라!"**

Rebase는 겉보기엔 똑같아 보여도 내부적으로 커밋의 고유 해시값(주민등록번호)을 아예 새로 발급하여 과거 히스토리를 재창조하는 작업입니다.
이미 원격 브랜치에 코드가 올라갔고 다른 팀원이 그걸 받아 사용 중인데 내가 로컬에서 몰래 Rebase로 히스토리를 세탁해 버리면, 동료들의 Git은 "내가 알던 과거와 다르다"며 엄청난 충돌 에러를 뿜어냅니다.

#### 💡 포트폴리오 안전 가이드
1. **내 로컬 브랜치**에서 자잘한 기능을 개발하는 동안에는 맘껏 커밋을 쪼개서 자주 기록하세요. (이게 훨씬 안전합니다.)
2. 원격 저장소에 PR을 올리기 **직전에**, 로컬에서 `rebase -i`를 사용해 커밋들을 예쁘게 하나의 기능 단위로 합칩니다.
3. 그 후, **깔끔하게 정돈된 단 하나의 커밋만 원격에** `push`하고 PR을 올립니다.
4. 만약 이미 원격 브랜치에 푸시한 내 개인 브랜치를 Rebase 했다면, 나 혼자 쓰는 브랜치에 한해서 `git push origin 브랜치명 --force` 명령어로 원격 저장소에 강제 동기화해야 합니다. (이때도 main 브랜치나 공유 브랜치에는 절대 사용하면 안 됩니다!)

## 🎯 오늘 배운 내용 요약
1. **Squash**는 자잘하고 부끄러운 여러 개의 중간 과정 커밋들을 단 하나의 의미 있는 단위 커밋으로 합치는 작업이다.
2. `git rebase -i HEAD~N` 명령어를 통해 과거 $N$개의 커밋들을 한눈에 보며 조작할 수 있다.
3. 가장 오래된 첫 번째 커밋만 `pick`으로 두고, 나머지 아래 커밋들은 `squash` (또는 `s`)로 설정해 하나로 뭉뚱그린다.
4. 원격 저장소에 공유되지 않은 **나만의 로컬 브랜치에서만 Rebase를 수행하는 것**이 실무의 안전 철칙이다.

배포 자동화: GitHub Actions로 CI/CD 구축하기 🤖

안녕하세요! 지난 5편에서 우리는 GitHub Pages를 사용해 로컬 폴더에 갇혀 있던 포트폴리오를 드디어 세상 밖으로 꺼내는 데 성공했습니다. 전 세계 누구나 내 주소를 치고 들어올 수 있게 되었죠.

하지만 사이트를 운영하다 보면 사소한 오타 하나를 고칠 때마다 매번 빌드하고, 푸시하고, 배포가 완료될 때까지 기다려야 하는 과정이 슬슬 귀찮아지기 마련입니다.

"내가 코드를 수정하고 `git push`만 하면, 누군가 알아서 빌드하고 배포까지 해줄 수는 없을까?"

이 귀찮음이 만든 현대 소프트웨어 공학의 정수가 바로 CI/CD(지속적 통합/지속적 배포)입니다. 오늘은 GitHub에서 제공하는 강력한 자동화 도구인 **GitHub Actions**를 사용해, 우리 포트폴리오에 자동 배포 파이프라인을 구축해 보겠습니다.

## 1. CI/CD가 도대체 무엇인가요? ⚙️

거창한 용어 같지만, 개발자의 귀찮음을 해결하고 실수를 방지하는 시스템입니다.
- **CI (Continuous Integration - 지속적 통합)**: 개발자가 작성한 코드를 중앙 저장소에 푸시하면, 자동으로 코드가 합쳐지고, 빌드가 잘 되는지 테스트하며 오류를 잡아내는 과정입니다.
- **CD (Continuous Deployment - 지속적 배포)**: CI 단계를 무사히 통과한 검증된 코딩 결과물을 실제 사용자가 접속하는 서버에 자동으로 올려 배포하는 과정입니다.
실무에서는 수십 명의 개발자가 매일 수백 번씩 코드를 고칩니다. 그때마다 사람이 직접 손으로 빌드하고 서버에 올린다면 휴먼 에러가 발생할 확률이 100%에 수렴하겠죠. 그래서 "빌드와 배포는 로봇(CI/CD 도구)에게 맡기고, 개발자는 오직 코드에만 집중한다"가 실무의 철칙입니다.

## 2. GitHub Actions: "GitHub에서 일하는 무료 비서 👮‍♂️"
GitHub Actions는 GitHub 저장소 안에서 특정 이벤트(예: `main` 브랜치에 코드 푸시)가 발생하면, **미리 약속된 명령어들을 클라우드 컴퓨터에서 자동으로 실행해 주는 도구**입니다.

GitHub Actions를 이해하기 위한 핵심 개념 4가지입니다.
1. **Workflow (워크플로우)**: 자동화할 전체 작업 흐름입니다. (예: "포트폴리오 배포 프로세스")
2. **Event (이벤트)**: 워크플로우를 실행시키는 방화쇠입니다. (예: `main` 브랜치에 `push` 되었을 때)
3. **Runner (러너)**: GitHub가 제공하는 가상의 컴퓨터(가상머신)입니다. 이 컴퓨터 안에서 우리의 빌드 코드가 돌아갑니다.
4. **Step & Action (단계와 액션)**: 가상머신 안에서 실행할 구체적인 명령어들의 집합입니다. (예: 레포지토리 복사하기 -> 빌드하기 -> 배포하기)

## 3. 실전! GitHub Actions 워크플로우 작성하기 🪄
우리의 포트폴리오 저장소에 자동화 명령서를 전달해 봅시다.

#### 1단계: 설정 파일 생성
프로젝트 루트 폴더에 `.github` 폴더를 만들고, 그 안에 `workflows` 폴더를 만든 뒤, `deploy.yml` 파일을 생성합니다. (경로가 정확해야 GitHub 로봇이 인식할 수 있습니다!)
```
📂 파일 경로: .github/workflows/deploy.yml
```

#### 2단계: 최신 표준 YAML 스크립트 작성
아래의 코드를 `deploy.yml` 파일에 붙여넣습니다. 이 코드는 GitHub이 공식 권장하는 최신 정적 사이트 배포 템플릿입니다.
```
name: Deploy Portfolio to Pages

on:
  # main 브랜치에 push가 발생하면 워크플로우 실행
  push:
    branches: ["main"]
  # GitHub 웹사이트에서 수동으로도 배포를 실행할 수 있게 허용
  workflow_dispatch:

# 배포를 위해 필요한 권한 설정 (GitHub Actions가 Pages 서버에 접근할 수 있도록 부여)
permissions:
  contents: read
  pages: write
  id-token: write

# 배포가 동시에 여러 번 돌아서 꼬이는 것을 방지
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    steps:
      # 1. GitHub 가상 컴퓨터로 내 포트폴리오 코딩 파일들을 복사해 옵니다.
      - name: Checkout
        uses: actions/checkout@v4

      # 2. GitHub Pages 배포를 위한 기본 환경 설정을 수행합니다.
      - name: Setup Pages
        uses: actions/configure-pages@v5

      # 3. 배포할 에셋(파일)들을 묶어서 임시 저장소(Artifact)에 올립니다.
      #    * 만약 React/Vite 등을 사용하여 빌드 결과물(dist/)을 배포해야 한다면
      #      path: './dist' 로 수정하세요. 기본 HTML/CSS 라면 '.' (루트)로 둡니다.
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: '.'

      # 4. 임시 저장소에 올라간 에셋을 실제 GitHub Pages 서버에 배포합니다.
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## 4. GitHub Pages 배포 방식 변경하기 (중요! ⚠️)
GitHub Actions를 통해 배포하게 하려면, 기존의 브랜치 수동 배포 설정을 **Actions 연동 방식**으로 바꿔주어야 합니다.

1. 내 GitHub 포트폴리오 저장소(`my-portfolio`) 페이지로 이동합니다.
2. 상단 탭에서 \[Settings\] -> 왼쪽 메뉴에서 \[Pages\]를 클릭합니다.
3. **Build and deployment** 섹션의 **Source** 항목을 확인합니다.
4. 기존의 `Deploy from a branch`를 클릭하여 `GitHub Actions`로 변경해 줍니다.

## 5. 자동화 마법의 결과 확인하기 🚀
이제 터미널을 열고, 우리가 만든 설정 파일을 저장소에 푸시해 봅시다.
```
# 1. 변경된 .github 폴더 담기
git add .github/

# 2. 커밋 남기기 (컨벤션 준수!)
git commit -m "chore: GitHub Actions를 활용한 자동 배포 CI/CD 파이프라인 구축"

# 3. 원격 저장소로 발사!
git push origin main
```

푸시가 끝난 뒤 GitHub 사이트로 접속하여 **\[Actions\]** 탭을 클릭해 보세요.

우리가 만든 `Deploy Portfolio to Pages` 워크플로우가 자동으로 실행되어 노란색 불이 깜빡거리며 열심히 빌드 작업을 수행하고 있는 모습을 볼 수 있습니다. 1~2분 뒤, 모든 과정이 끝나고 영롱한 초록색 체크 마크(SUCCESS)가 뜨면 배포 자동화가 완료된 것입니다!

이제 여러분의 작업 루틴은 다음과 같이 완전히 바뀝니다.
- **코드를 고친다 👉 git push를 한다 👉 끝! (나머지는 Actions 비서가 알아서 배포 완료)**

## 🎯 오늘 배운 내용 요약
1. **CI/CD**는 개발자의 단순 반복적인 빌드 및 배포 작업을 자동화하여 휴먼 에러를 막고 생산성을 높이는 필수 기술이다.
2. **GitHub Actions**는 우리가 정의한 스크립트(`.github/workflows/deploy.yml`)를 기반으로 가상 컴퓨터에서 명령을 대신 수행해 준다.
3. 내 손으로 직접 배**포 자동화 파이프라인**을 구축하여 주소에 즉시 반영되는 진정한 '자동 배포의 맛'을 경험했다.

이제 여러분의 포트폴리오 저장소에는 단순 소스 코드뿐만 아니라, **인프라 자동화(DevOps) 설정 파일**까지 탑재되었습니다. 이 저장소를 열어보는 시니어 개발자들은 "이 지원자는 DevOps와 배포 자동화의 흐름까지 이해하고 있구나"라며 엄청난 가산점을 부여할 것입니다.

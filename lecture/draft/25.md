자동화 경찰관 배치: Husky와 Linter 연동 👮‍♂️🛠️

안녕하세요! 지난 7편에서는 Interactive Rebase를 통해 과거의 부끄러운 커밋 히스토리를 한 점 부끄럼 없이 깨끗하게 세탁하는 마법을 배웠습니다.

하지만 매번 커밋을 하고 난 후에 Rebase로 정리하거나, 커밋을 하기 전에 "오타가 있나?", "코드 스타일이 깨졌나?" 하고 일일이 손으로 검사하는 것은 무척 피곤한 일입니다. 사람이 하는 일에는 언제나 실수가 따르기 마련이니까요. 피곤한 날에는 실수로 console.log를 남겨두거나, 세미콜론을 빼먹거나, 정렬이 엉망인 코드를 그대로 커밋하기 쉽습니다.

실무에서는 이를 예방하기 위해 **에러 방지 자동화 시스템**을 구축합니다. 사람이 조심하는 것이 아니라, 규칙에 맞지 않는 코드는 애초에 `git commit` **자체가 불가능하도록 입구에서 원천 봉쇄**하는 것이죠.

오늘은 우리의 포트폴리오 프로젝트에 엄격하고 완벽한 철통 보안을 담당할 자동 검사 경찰관, Husky와 Lint-staged를 배치해 보겠습니다!

## 1. Git Hooks와 Husky란 무엇인가요? 🪝
Git에는 특정 이벤트(예: 커밋 직전, 푸시 직전 등)가 발생할 때 사용자가 지정한 스크립트를 자동으로 실행해 주는 Git Hooks 기능이 내장되어 있습니다.

그중에서 우리가 사용할 핵심 이벤트는 바로 `pre-commit`입니다. 이름 그대로 커밋 메시지를 적기 직전에 실행되는 훅으로, 이때 코드 검사를 수행하여 통과하지 못하면 커밋 프로세스를 강제로 중단(Abort)시킵니다. 즉, 오류 발생률을 $0%$에 가깝게 수렴시키는 강력한 방어선입니다.

#### ❓ 내장 기능이 있는데 왜 굳이 'Husky'를 쓰나요?
기본 내장 Git Hooks는 프로젝트의 `.git/hooks` 폴더 안에 저장됩니다. 문제는 이 `.git` 폴더가 `.gitignore`의 대상이자 Git의 내부 저장 공간이기 때문에, **GitHub 원격 저장소에 올라가지 않는다**는 점입니다.

즉, 내가 고생해서 만든 멋진 검사 스크립트를 동료 개발자나 미래의 내가 다른 컴퓨터에서 `clone` 받았을 때 자동으로 공유할 수 없습니다.

이 문제를 깔끔하게 해결해 주는 도구가 바로 Husky(허스키)입니다. Husky는 Git Hooks 설정을 패키지 설정 파일(`package.json`) 및 별도의 디렉토리(`.husky/`)로 관리할 수 있도록 꺼내어 주어, **Git을 통해 팀원들과 훅 설정을 완벽하게 공유**할 수 있게 해줍니다.

## 2. ⚡ 효율성을 극대화하는 비서, lint-staged
"커밋하기 전에 코드 스타일 검사를 실행한다."

단순히 생각하면 Husky에 `npm run lint`나 `npm run test`를 지정해두면 될 것 같습니다. 하지만 프로젝트의 규모가 점점 커져서 소스 파일이 수천 개가 된다면 어떻게 될까요?

겨우 파일 한 줄 고치고 커밋을 하려는데, 매번 전체 프로젝트 파일 수천 개를 검사하느라 커밋 한 번 할 때마다 수십 초에서 몇 분씩 기다려야 하는 대참사가 일어납니다. 개발 효율이 극도로 떨어지겠죠.

이 답답함을 해결해 주는 똑똑한 파트너가 바로 **lint-staged**입니다.

lint-staged는 전체 파일을 검사하는 것이 아니라, **현재 Staging Area에 올라와 있는(즉, 내가 이번에 `git add`로 담은) 파일들만 골라내어 검사를 실행**해 줍니다.

덕분에 전체 파일 개수가 N개인 대형 프로젝트라 할지라도, 내가 수정한 1~2개의 파일에 대해서만 초고속으로 검사를 수행하므로 시간 복잡도를 혁신적으로 줄여줍니다.

## 3. 🛠️ 실전! Husky + Lint-staged 완벽 세팅하기
그럼 우리 프로젝트(Node.js 기반 환경 가정)에 두 경찰관을 직접 고용해 봅시다.

#### 1단계: 패키지 설치
터미널을 열고 프로젝트 폴더에서 아래 패키지들을 개발용 의존성(`-D`)으로 설치합니다.
```
npm install -D husky lint-staged
```

#### 2단계: Husky 초기화 및 연동
설치가 끝났다면, Husky가 Git Hooks를 제어할 수 있도록 초기화 명령을 내립니다. 최신 표준 방식인 다음 명령어를 실행합니다.
```
npx husky init
```

이 명령어를 실행하면 다음과 같은 마법 같은 일들이 일어납니다.

1. 프로젝트 루트에 `.husky/` 폴더가 자동으로 생성됩니다.
2. `package.json` 파일의 `scripts` 영역에 `"prepare": "husky"`라는 명령어가 자동으로 등록됩니다. (이제 프로젝트를 `clone` 받은 다른 사람들도 `npm install`만 하면 자동으로 Husky가 설치 및 활성화됩니다.)
3. `.husky/pre-commit` 샘플 파일이 자동으로 생성됩니다.

## 3단계: lint-staged 설정 추가하기
이제 `package.json` 파일에 어떤 파일 확장자를 어떤 도구(Linter, Prettier 등)로 검사할지 명세서를 작성해 줍니다.

`package.json` 파일의 가장 아래쪽에 아래 내용을 추가해 주세요.
```
{
  ...
  "scripts": {
    ...
  },
  "lint-staged": {
    "*.{js,jsx,ts,tsx}": [
      "eslint --fix",
      "prettier --write"
    ],
    "*.{css,json,md}": [
      "prettier --write"
    ]
  }
}
```

- `"eslint --fix"`: 코드 분석 도구인 ESLint를 사용해 문법 오류를 잡고, 자동으로 고칠 수 있는 경미한 룰(예: 안 쓰는 따옴표 정리 등)은 즉시 수정합니다.
- `"prettier --write"`: 코드 포맷터인 Prettier를 사용해 들여쓰기, 세미콜론 등의 스타일을 일관되게 강제 자동 정돈합니다.

## 4단계: pre-commit 훅 파일 수정하기
이제 커밋 직전에 방금 설정한 `lint-staged`가 실행되도록 연결해 줍니다. `.husky/pre-commit` 파일을 열고 내용을 아래와 같이 수정합니다.
```
#!/bin/sh
. "$(dirname "$0")/_/husky.sh"

npx lint-staged
```

## 4. 🚨 경찰관의 실전 단속 현장 확인하기 (테스트)
자, 세팅이 끝났으니 검문소가 제대로 작동하는지 고의로 나쁜 코드를 짜서 테스트해 봅시다!

#### 🧪 일부러 엉망인 코드 작성하기
아무 파일(`index.js` 등)이나 열어서 들여쓰기를 무시하고 지저분하게 코드를 작성해 봅니다.
```
// index.js
function test() {
const unusedVar = 'I am not used!'; // 선언 후 쓰지 않는 변수 (ESLint 에러 유도)
  console.log("들여쓰기도 엉망이고 안 쓰는 변수도 있죠")
}
```

이제 이 파일을 저장하고, 카트에 담은 뒤 커밋을 시도해 봅니다.
```
git add index.js
git commit -m "feat: 자동화 단속 테스트"
```

#### 🔍 결과 확인
명령어를 치면 터미널에 다음과 같이 로그가 뜨며 단속이 시작됩니다.
```
✔ Preparing lint-staged...
❯ Running tasks for staged files...
  ❯ package.json — 1 file
    ✔ Run eslint --fix
    ✔ Run prettier --write
```

만약 단순한 들여쓰기 오류였다면 Prettier가 알아서 눈 깜짝할 사이에 코드를 `prettier --write`**를 통해 예쁘게 자동 수정해주고 커밋까지 완료**시킵니다. 에디터를 확인해 보면 들여쓰기가 마법처럼 칼같이 맞춰져 있는 것을 볼 수 있습니다.

하지만 ESLint의 `no-unused-vars`(안 쓰는 변수 금지)처럼 **자동으로 해결할 수 없는 치명적인 문법 에러**가 있다면 아래와 같이 에러가 발생합니다.
```
✖ eslint --fix:
  /src/index.js
    2:7  error  'unusedVar' is assigned a value but never used  no-unused-vars

✖ eslint --fix jst [FAILED]
husky - pre-commit hook exited with code 1 (error)
```

그리고 `git commit` **프로세스 자체가 통째로 취소**됩니다! 지저분하거나 고장 난 버그 코드가 내 저장소 히스토리에 진입하는 것을 완벽하게 막아낸 순간입니다. 이제 개발자는 에러 메시지를 보고 코드를 올바르게 수정한 뒤에야 기분 좋게 커밋을 완료할 수 있게 됩니다.

## 🎯 오늘 배운 내용 요약
1. **Git Hooks**의 `pre-commit` 이벤트를 이용하면 문법 검사나 코드 포맷팅을 완벽하게 자동화할 수 있다.
2. **Husky**는 로컬 컴퓨터에만 머무는 Git Hooks 설정을 원격 저장소에 올려 팀원들과 공유할 수 있게 돕는다.
3. **lint-staged**는 오직 `git add` 된 변경 영역 파일들만 빠르게 검사하여 대규모 프로젝트에서도 버벅임 없이 성능을 극대화한다 ($O(1)$의 효율성).
4. 시스템적으로 실수를 차단하는 파이프라인을 다룰 줄 아는 역량은 시니어 개발자들에게 엄청난 신뢰감을 준다.

이제 여러분의 포트폴리오는 깔끔한 커밋 히스토리를 넘어, 그 누구도 더러운 코드를 올릴 수 없게 설계된 '실무형 자동 제어 시스템'을 품게 되었습니다!

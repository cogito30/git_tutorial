$$실무 Git 튜토리얼 11편: 비기$$

지워진 커밋도 살려내는 마법: Reflog 🧙‍♂️

안녕하세요! 지난 10편에서는 여러 가지 실수 상황을 되돌리는 방법을 배웠습니다.

하지만 지난번 튜토리얼 마지막에 남겼던 무시무시한 떡밥을 기억하시나요?
밤샘 코딩의 피로에 찌들어 몽롱한 상태에서 터미널에 그만 `git reset --hard HEAD~3`을 쳐버렸습니다. 무려 3번의 커밋, 3일 치 코드가 내 컴퓨터에서 흔적도 없이 사라져 버린 것이죠. `git log`를 쳐봐도 과거 기록은 이미 날아가고 없습니다.

등줄기에 식은땀이 흐르고 사표를 써야 하나 고민되는 순간, Git은 우리에게 마지막 구원의 밧줄을 내려줍니다. 바로 오늘 배울 마법의 명령어, `Reflog`입니다.

## 1. Reflog란? "Git의 숨겨진 CCTV 📹"
`git log`가 우리가 명시적으로 남긴 '영수증(커밋 기록)'을 보여준다면, `git reflog` (Reference log)는 내 컴퓨터 Git의 'CCTV 카메라 영상'을 보여줍니다.

여러분(HEAD)이 브랜치를 이동하고, 커밋을 하고, 심지어 `reset`으로 커밋을 지워버린 **모든 행적을 초 단위로 빠짐없이 기록해 둔 숨겨진 블랙박스**입니다.

커밋이 지워졌다고 해도 진짜 영구 삭제된 것이 아니라, 화면(log)에서만 안 보이게 숨겨진 채 이 블랙박스 안에 고스란히 남아있습니다.

## 2. 🪄 실전: 날아간 3일 치 코드 부활시키기
자, 심호흡을 하고 날아간 코드를 되살려 봅시다.

#### 🔍 Step 1. 블랙박스 열어보기: `git reflog`
터미널에 다급하게 아래 명령어를 칩니다.
```
git reflog
```
그러면 아래와 같이 내가 그동안 Git에서 했던 모든 행동이 외계어처럼 나타납니다.
```
7a8b9c0 (HEAD -> main) HEAD@{0}: reset: moving to HEAD~3  <-- [내가 방금 친 치명적인 실수!!]
1d2e3f4 HEAD@{1}: commit: feat: 결제 모듈 최종 연동      <-- [이게 날아간 3일 차 코드!]
5g6h7i8 HEAD@{2}: commit: feat: 장바구니 로직 수정      <-- [이건 날아간 2일 차 코드!]
9j0k1l2 HEAD@{3}: commit: feat: 상품 목록 UI 개선        <-- [이건 날아간 1일 차 코드!]
7a8b9c0 HEAD@{4}: commit: fix: 로그인 버그 수정
...
```
가장 위쪽(`HEAD@{0}`)이 방금 내가 저지른 끔찍한 `reset`기록입니다.
그 바로 아래 `HEAD@{1}`을 보니, 아까까지만 해도 살아있던 '**결제 모듈 최종 연동**' 커밋이 보입니다! 이 녀석의 고유 번호(Hash)는 `1d2e3f4`네요.

#### 🔄 Step 2. 타임머신 타고 과거로 가기: 다시 `reset`
아이러니하게도, `reset`으로 망친 코드를 살려내는 방법 역시 `reset`입니다.
우리가 알아낸 '사고 나기 직전의 커밋 번호'를 향해 다시 타임머신을 타면 됩니다.
```
# 사고 나기 직전의 커밋 번호(또는 HEAD@{1})로 강제 복구!
git reset --hard 1d2e3f4 
# (또는 git reset --hard HEAD@{1} 이라고 쳐도 됩니다)
```
엔터를 치는 순간! 에디터에서 싹 사라졌던 3일 치 코드들이 마법처럼 다시 촤르륵 나타납니다. 🎉 `git log`를 다시 쳐보면 지워졌던 기록들도 완벽하게 돌아와 있습니다.

## 3. 💡 응용: 실수로 삭제한 브랜치도 살릴 수 있나요?
물론입니다! `main`브랜치에 코드를 합치지도 않았는데, 실수로 `git branch -D feature/login`을 쳐서 며칠 동안 작업한 브랜치를 통째로 날려버렸을 때도 똑같습니다.
1. `git reflog`를 쳐서 날아간 브랜치의 마지막 커밋 번호를 찾습니다.
2. 그 번호를 이용해 지워진 브랜치를 다시 만들어주면 끝입니다!
```
# 복구할커밋번호에 해당하는 상태로 'feature/login' 브랜치를 다시 생성!
git branch feature/login 복구할커밋번호
```

## 4. 🚨 Reflog의 한계 (주의사항)
이렇게 사기적인 능력을 가진 Reflog도 만능은 아닙니다. 다음 두 가지는 꼭 명심하세요.

1. **커밋(또는 Stash)을 한 번이라도 한 적이 있어야 합니다**. `git add`조차 안 하고 작성 중이던 쌩 초보 상태의 파일(Untracked file)을 에디터에서 지워버리거나 `reset --hard`로 날렸다면, Git의 CCTV에도 찍히지 않았기 때문에 영영 살릴 수 없습니다. (이래서 작업을 세세하게 쪼개서 자주 commit 하는 습관이 중요합니다!)
2. **유통기한이 있습니다**. Reflog는 로컬(내 컴퓨터)에만 저장되는 찌꺼기 파일 같은 개념이라, 보통 90일이 지나면 Git이 용량 확보를 위해 영구적으로 삭제해 버립니다. (물론 90일 전 실수를 되돌릴 일은 거의 없겠지만요!)

## 🎯 오늘 배운 내용 요약 (최후의 보루)
1. `reset --hard`로 코드를 날렸거나 브랜치를 삭제했다면 당황하지 말고 `git reflog`를 치자.
2. Git의 블랙박스를 보고 사고 나기 직전의 커밋 번호(Hash)를 찾는다.
3. `git reset --hard 복구할커밋번호`로 되살려 낸다!

이제 여러분은 Git에서 발생하는 웬만한 휴먼 에러는 모두 스스로 복구할 수 있는 진정한 강심장을 가지게 되었습니다!

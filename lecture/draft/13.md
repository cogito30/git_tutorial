 휴먼 에러를 막아라: Git Hooks (Husky) 🤖

안녕하세요! 어느덧 대망의 마지막 시간입니다. 지난 12편까지 우리는 Git 명령어 활용법부터 브랜치 전략까지, 실무에 필요한 모든 기술적 요소를 섭렵했습니다.

하지만 실무에서 가장 무서운 것은 무엇일까요? 바로 '사람'입니다.
피곤한 오후, 코드를 수정하고 실수로 eslint 에러를 방치한 채 커밋을 하거나, 테스트 코드를 돌리지 않고 푸시를 해서 서비스 전체를 멈추게 하는 아찔한 경험... 누구나 한 번쯤 겪는 휴먼 에러입니다.

오늘 배울 Git Hooks는 여러분의 팀을 이런 사고로부터 지켜줄 든든한 '자동화 경찰관'입니다.

1. Git Hooks란? "커밋을 지켜보는 경찰관 👮‍♂️"

Git Hooks는 Git의 특정 사건(이벤트)이 발생할 때마다 자동으로 특정 코드를 실행하게 해주는 기능입니다.

pre-commit: 커밋을 날리기 직전!

pre-push: 서버에 푸시하기 직전!

이 타이밍에 우리가 미리 지정해 둔 검사 도구(Linter, 테스트 코드 등)를 자동으로 실행시켜서, 만약 에러가 하나라도 발견되면 커밋이나 푸시 자체를 거부해버리는 것이죠.

2. 🪄 Husky: Hook 설정을 돕는 마법사

원래 Git Hooks를 설정하려면 복잡한 스크립트를 직접 짜야 해서 번거롭습니다. 그래서 요즘 실무에서는 Husky라는 라이브러리를 사용해 아주 쉽게 관리합니다.

🛠️ Step 1. 설치 및 자동화 (Node.js 환경 예시)

# Husky 설치 및 초기화
npm install husky --save-dev
npx husky install

# 커밋하기 전에 실행할 명령어 등록 (예: lint 검사)
npx husky add .husky/pre-commit "npm run lint"



이렇게 설정해두면, 누군가 git commit을 하는 순간, Husky가 자동으로 npm run lint를 실행합니다. 코드 스타일에 문제가 있으면 커밋이 아예 안 됩니다!

3. 실무 꿀팁: 우리 팀을 위한 '강철 보안' 만들기

실무에서는 보통 이런 조합을 가장 많이 씁니다.

pre-commit: 코드를 저장할 때 ESLint(코드 스타일 검사)와 Prettier(코드 포맷팅)를 돌립니다. "지저분한 코드는 절대 내 Git 저장소에 들어올 수 없다!"는 강력한 의지죠.

pre-push: 원격 서버에 보내기 직전, jest 같은 테스트 코드를 전면 실행합니다. 테스트가 통과되지 않으면? 절대로 서버에 코드가 올라가지 않습니다.

이렇게 해두면 팀원들이 실수로 에러 코드를 올릴 걱정이 사라지므로, 모두가 안심하고 개발할 수 있는 환경이 만들어집니다.

🎯 대단원의 막을 내리며: 여러분의 Git 여정

지난 1편부터 여기까지, 정말 긴 여정이었습니다.

Git 입문: 터미널과 환경설정

기본 사이클: add, commit, log

브랜치와 병합: switch, merge, 충돌 해결

협업의 꽃: Remote 저장소와 Pull Request(PR)

실무 고수 스킬: Stash, Rebase, Cherry-pick

위기 복구: Reset, Revert, Reflog

팀 문화: Branching Strategies, Git Hooks

이제 여러분은 단순히 명령어를 외우는 초보자를 넘어, 팀의 코드를 보호하고, 히스토리를 관리하며, 실수를 시스템으로 막아낼 줄 아는 '어엿한 실무 개발자'가 되었습니다.

Git은 단순히 코드를 저장하는 툴이 아닙니다. 팀원들과 소통하는 언어이며, 여러분의 개발 실력을 촘촘하게 기록해 주는 자산입니다. 앞으로 여러분의 GitHub 잔디밭이 더 푸르게 빛나길 진심으로 응원합니다!

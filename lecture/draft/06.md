실무 협업의 꽃, Pull Request (PR) 🌸

안녕하세요! 지난 5편에서 드디어 코드를 인터넷(GitHub)에 올리고 내려받는 방법을 배웠습니다.

이제 여러분은 회사에 출근해서 코드를 수정하고 `git push origin main`을 힘차게 입력합니다! ...잠깐, 사수님이 달려오네요.
```
👨‍💻 사수: "A씨!! 지금 라이브 서비스 중인 `main` 브랜치에 직접 푸시하시면 어떡해요! 서비스 터져요!! 기능 다 만들었으면 PR을 올려주셔야죠!"
```
네, 그렇습니다. 실무에서는 여러 명이 함께 코드를 다루기 때문에, 아무나 원본 코드(main)를 마구잡이로 수정하게 내버려 두지 않습니다. 이때 등장하는 안전장치이자 훌륭한 협업 문화가 바로 오늘 배울 Pull Request(PR)입니다.

1. Pull Request (PR)란 무엇인가요? 🙋‍♂️

단어 뜻 그대로 해석해 보면 "내가 작업한 브랜치(Pull)를 원본 브랜치에 당겨가서 합쳐달라고 요청(Request)하는 것"입니다.

나: "사수님, 저 feature/login 브랜치에서 로그인 기능 다 만들어서 GitHub에 올려뒀어요. 코드 한 번 검사해 보시고, 문제없으면 main 브랜치에 합쳐주세요!"

이 과정을 GitHub 웹사이트에서 버튼을 눌러 시각적인 게시물 형태로 만드는 것이 바로 PR입니다. PR을 올리면 팀원들은 내 코드를 줄 단위로 읽어보며 피드백을 주고받는 '코드 리뷰(Code Review)'를 진행하게 됩니다.

2. 🔄 [실무 필수] 팀 개발의 표준 워크플로우

실무에서 하나의 기능이 개발되어 반영되기까지의 표준적인 흐름을 알아봅시다. 이 흐름만 알아도 당장 현업에 투입될 수 있습니다!

이슈(Issue) 할당: 해야 할 작업이나 버그 수정 목표가 주어집니다.

브랜치 생성 및 이동: main에서 git switch -c feature/login

작업 및 커밋: 코드를 열심히 짜고 git add & git commit

원격 저장소에 푸시: 내 브랜치 그대로 인터넷에 올리기 👉 git push origin feature/login

⭐ PR 생성: GitHub에 들어가서 "이 브랜치 합쳐주세요!" 하고 PR 게시글을 씁니다.

코드 리뷰 (Code Review): 팀원들이 내 코드를 보고 "이 부분은 이렇게 수정하는 게 좋겠어요"라고 댓글을 답니다.

Merge (병합): 수정이 완료되고 팀원들의 승인(Approve)을 받으면, 최종적으로 관리자가 코드를 main에 합칩니다!

3. 💌 사랑받는 주니어의 '좋은 PR' 작성법

PR은 단순히 "다 했어요"라고 올리는 것이 아닙니다. 리뷰어(내 코드를 읽는 동료)가 내 코드를 쉽게 이해할 수 있도록 친절하게 설명서를 적어주어야 합니다.

실무에서 자주 쓰는 PR 템플릿(양식) 구조는 다음과 같습니다.

✅ 좋은 PR 본문 예시

## 📝 작업 내용 (무엇을 했나요?)

카카오 소셜 로그인 기능 추가

로그인 버튼 UI 수정 (파란색 -> 노란색)

## 📸 스크린샷 (선택 사항)
(변경 전/후 화면 캡처나 작동하는 GIF 화일을 첨부하면 사랑받습니다 💛)

## 💡 리뷰어에게 당부하는 말 (어디를 중점적으로 볼까요?)

login.js의 34번째 줄 로직이 최선인지 헷갈리는데, 더 좋은 방법이 있을까요?

버튼 컴포넌트 분리했는데 네이밍이 괜찮은지 봐주세요!

이렇게 PR을 올리면, 사수님은 내 질문에 맞춰 더 퀄리티 높은 피드백을 줄 수 있습니다.

4. 💡 실무 용어 알아듣기: "LGTM!" 👍

PR을 올리고 코드 리뷰를 받다 보면, 팀원들이 댓글로 "LGTM"이라고 남기고 승인(Approve)을 해주는 경우가 많습니다.

이것은 "Looks Good To Me (내 보기엔 좋네요, 합쳐도 되겠습니다)"의 약자입니다. 실무 개발자들이 가장 사랑하는 마법의 단어죠! 이 댓글이 달렸다면 안심하고 코드를 Merge 하셔도 좋습니다.

5. [참고] 오픈소스는 어떻게 기여할까? (Fork)

사내 프로젝트가 아니라, 전 세계인이 다 함께 만드는 오픈소스(예: React, Vue 등)에 기여하고 싶다면 어떻게 할까요? 나는 그 회사의 직원이 아니기 때문에 마음대로 브랜치를 만들 권한(Push 권한)이 없습니다.

이때 사용하는 것이 Fork(포크로 찍어오기)입니다.

Fork: 원본 프로젝트를 통째로 '내 GitHub 계정'으로 복사해 옵니다.

작업: 내 계정에 복사된 저장소에서 코드를 고치고 푸시합니다. (내 거니까 맘대로 가능!)

PR: "내 계정에 있는 수정본"을 "원본 프로젝트"에 합쳐달라고 PR을 보냅니다.

오픈소스 생태계는 이런 방식으로 누구나 코드를 수정해서 제안할 수 있도록 열려 있습니다.

🎯 오늘 배운 내용 요약

Pull Request(PR)는 내 코드를 main에 합치기 전, 팀원들에게 검사와 리뷰를 요청하는 과정이다.

브랜치를 내 컴퓨터에서 파고 코딩한 뒤, 그 브랜치 이름 그대로 push 해야 PR을 열 수 있다.

PR 본문을 친절하게 적고 스크린샷을 첨부하면 리뷰어에게 사랑받는다.

권한이 없는 남의 프로젝트(오픈소스)에 PR을 보낼 때는 Fork를 사용한다.

이제 여러분은 내 코드를 올리고, 팀원들과 소통하며 프로젝트를 완성해 나가는 어엿한 '실무형 깃 사용자'가 되었습니다! 👏👏👏

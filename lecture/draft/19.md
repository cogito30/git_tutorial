프로의 기본기: .gitignore와 커밋 컨벤션 세팅하기 🧹

안녕하세요! 지난 1편에서 포트폴리오 프로젝트의 기획을 마치고 로컬 저장소 초기화(git init)까지 무사히 마쳤습니다. 이제 코딩을 시작하면 될까요?

잠시만요! 본격적으로 한 줄의 코드를 작성하기 전에, **프로 개발자라면 무조건 세팅하고 시작하는 두 가지 기본기**가 있습니다. 바로 `.gitignore` 설정과 **커밋 컨벤션(Commit Convention)** 수립입니다.

이 두 가지는 단순히 기술적인 설정을 넘어, 채용 담당자가 여러분의 GitHub 저장소를 방문했을 때 "이 지원자는 기본기가 아주 탄탄하고, 당장 실무에 투입해도 협업에 문제가 없겠구나"라는 강한 첫인상을 주는 결정적인 요소입니다.

## 1. 보안과 용량 관리의 첫 걸음, `.gitignore` 🔒
우리가 코딩을 시작하면 눈에 보이는 코드 파일 외에도 수많은 파일들이 자동으로 생성됩니다.

예를 들어, 패키지를 설치하면 생기는 엄청난 크기의 `node_modules/` 폴더, 운영체제가 자동으로 만드는 임시 파일(`.DS_Store`, `Thumbs.db`), 에디터 설정 파일(`.vscode/`), 그리고 무엇보다 절대 남들에게 보여주면 안 되는 API 키가 담긴 환경 변수 파일(`.env`) 등이 있습니다.

이런 파일들이 실수로 GitHub에 올라가면 다음과 같은 대참사가 발생합니다.
- **용량 폭발**: 무거운 외부 라이브러리들 때문에 `git clone`을 받는 데 한 세월이 걸립니다.
- **보안 사고**: 실수로 올린 AWS 키나 데이터베이스 패스워드를 해커들이 긁어가서 수천만 원의 요금 폭탄을 맞을 수 있습니다.
- **커밋 로그 오염**: 코드 한 줄 바꿨는데 빌드 파일 수백 개가 같이 커밋되어 어떤 코드가 바뀌었는지 알아볼 수 없게 됩니다.
이 모든 비극을 막아주는 마법의 장치가 바로 `.gitignore` 파일입니다.

#### 🛠️ 실무자처럼 `.gitignore` 세팅하기
프로젝트 루트 폴더(최상위 경로)에 `.gitignore`라는 이름의 파일을 만들고 아래 내용을 채워 넣습니다.

💡 **실무자 팁: gitignore.io 활용하기**
실무에서는 일일이 타이핑하지 않고, [toptal gitignore](https://www.toptal.com/developers/gitignore) 사이트를 애용합니다. 내가 사용하는 OS(macOS, Windows), 에디터(vscode), 기술 스택(React, Vite, Node 등)을 검색하면 완벽한 `.gitignore` 템플릿을 자동으로 생성해 줍니다.

아래는 포트폴리오 사이트 제작에 딱 맞춘 표준 `.gitignore` 템플릿입니다.
```
# --- Operating System Files ---
.DS_Store
Thumbs.db

# --- Editor / IDE Files ---
.vscode/
.idea/
*.suo
*.ntvs*
*.njsproj
*.sln

# --- Node / Package Manager ---
node_modules/
npm-debug.log*
yarn-debug.log*
yarn-error.log*
.pnpm-debug.log*

# --- Build Outputs ---
dist/
build/
out/

# --- Environment Variables (보안 최우선!) ---
.env
.env.local
.env.development.local
.env.test.local
.env.production.local

# --- Log Files ---
logs
*.log
```

#### 🚨 이미 올려버린 파일 무시하기 (응급처치)
만약 `.gitignore`를 쓰기 전에 실수로 `node_modules`나 임시 파일을 커밋해 버렸다면, 단순히 파일만 지우면 안 되고 Git의 추적 대상(캐시)에서 제외해 주어야 합니다.
```
# 로컬의 실제 파일은 놔두고, Git 추적(저장소)에서만 제거합니다.
git rm -r --cached node_modules/
git commit -m "chore: git 추적 대상에서 node_modules 제외"
```

## 2. 채용 담당자를 매료시키는 '커밋 컨벤션' 🏷️
포트폴리오의 품질을 평가할 때, 시니어 개발자들이 가장 먼저 열어보는 곳 중 하나가 바로 git log (커밋 히스토리)입니다.

만약 여러분의 저장소에 이런 커밋들만 가득하다면 어떨까요?
- `수정`
- `진짜 마지막 수정`
- `아 왜 안돼`
- `디자인 고침`

이런 로그는 "저는 협업 규칙을 모르고, 제 맘대로 코딩합니다"라고 광고하는 것과 다름없습니다. 반면, 아래처럼 일관되고 정돈된 메시지가 적혀 있다면 어떨까요?
- `feat`: 자기소개 컴포넌트 마크업 추가
- `fix`: 모바일 화면에서 내비게이션 바 깨짐 현상 수정
- `style`: 메인 컬러 테마 변경 (블루 -> 다크 그레이)
담당자는 코드를 열어보기도 전에 "이 사람은 기본기가 완성이 되어 있구나"라며 호감을 갖게 됩니다.

#### ✍️ 커밋 메시지 3대 구조
실무 표준으로 널리 쓰이는 커밋 메시지 규칙(Conventional Commits)은 다음과 같은 구조를 따릅니다.
```
타입(Type): 제목 (Header)
- 빈 줄 -
상세 설명 (Body) - 생략 가능
- 빈 줄 -
이슈 트래커 연동 (Footer) - 생략 가능
```

#### 🎯 반드시 외워야 할 핵심 커밋 타입 7가지
포트폴리오를 만들 때 가장 많이 쓰게 될 대표적인 말머리들입니다. 대괄호`[]`를 쓰거나 콜론`:` 뒤에 한 칸을 띄우는 형식을 주로 씁니다.

| 타입 (Type) | 언제 사용하나요? | 예시 | 
| :--: | :--: | :--: | 
| feat | 새로운 기능이나 요소를 추가할 때 | feat: 방명록 작성 기능 추가 | 
| fix | 버그나 에러를 수정할 때 | fix: 이메일 전송 실패 오류 해결
| docs | 문서만 수정할 때 (README, 주석 등) | docs: README.md 프로젝트 소개 업데이트
| style | 코드 로직 변경 없이 스타일, 세미콜론 누락, 포맷팅을 수정할 때 | style: 메인 레이아웃 패딩 값 조정
| refactor | 기능은 그대로 둔 채 코드를 가독성 좋게 개선할 때 | refactor: 카드 컴포넌트 구조 분할
| test | 테스트 코드를 추가하거나 수정할 때 | test: 연락처 폼 유효성 검사 테스트 추가
| chore | 패키지 빌드 설정, 라이브러리 설치 등 기타 자잘한 작업 | chore: Tailwind CSS 라이브러리 설치

💡 **나만의 약속 지키기**
혼자 진행하는 프로젝트라고 규칙을 어기지 마세요. 커밋을 날리기 전에 한 번 더 생각하고 규칙에 맞게 작성하는 훈련 자체가 포트폴리오의 가장 훌륭한 '스펙'이 됩니다.

## 🎯 오늘 배운 내용 요약
1. `.gitignore`는 프로젝트를 시작하자마자 가장 먼저 만드는 파일이다. 보안 위협과 불필요한 용량 낭비를 사전에 차단하자.
2. **커밋 컨벤션**은 내 작업 일지이자, 협업 역량을 보여주는 얼굴이다. 7가지 핵심 타입(`feat`, `fix`, `style` 등)을 꼭 지켜서 커밋을 작성하자.
환경 세팅과 규칙 수립이 끝났으니, 이제 날아오를 준비가 완료되었습니다!

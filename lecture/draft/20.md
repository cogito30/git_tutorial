기능별로 쪼개기: Feature Branch Flow 🌿

안녕하세요! 지난 2편에서 우리는 안전한 프로젝트 세팅을 위해 `.gitignore`를 설정하고, 협업의 얼굴인 커밋 컨벤션까지 완벽히 마쳤습니다.

이제 준비운동은 끝났습니다. 드디어 포트폴리오 사이트의 뼈대를 만들고 코드를 작성할 시간입니다!

그런데 잠깐, 혹시 아직도 `main`브랜치에 서서 곧바로 마우스 우클릭을 하고 `index.html` 파일을 만들 준비를 하고 계신가요? **실무에서는 절대로 `main` 브랜치에서 직접 개발하지 않습니다**.

오늘은 혼자 개발하더라도 시니어 개발자처럼 일하는 개발 프로세스의 핵심, Feature Branch Flow(기능별 브랜치 분리 전략)를 우리 포트폴리오에 이식해 보겠습니다.

## 1. "혼자 만드는데 브랜치를 왜 나누나요?" 🤔
가장 많이 받는 질문 중 하나입니다. 혼자 개발하면 충돌할 사람도 없고 내 마음대로 바로 합치면 편한데, 굳이 브랜치를 만들고 이동하고 합치는 과정이 번거롭게 느껴질 수 있습니다.

하지만 채용 담당자(리드 개발자)의 시선은 다릅니다. 그들이 여러분의 GitHub 저장소에서 가장 먼저 누르는 버튼 중 하나가 바로 **\[Insights\] -> \[Network\]** 탭입니다.

- **A 지원자의 네트워크 그래프**: `main` 브랜치 하나에서 모든 커밋이 일직선으로 쭉 이어져 있음.
- **B 지원자의 네트워크 그래프**: `main` 브랜치라는 든든한 기둥을 두고, `feature/about`, `feature/projects` 같은 브랜치들이 나뭇가지처럼 예쁘게 뻗어나갔다가 다시 합쳐지는 흐름이 시각적으로 보임.

담당자는 B 지원자의 그래프를 보는 순간 이렇게 판단합니다.
- "이 친구는 기능을 단위별로 쪼개서 작업할 줄 아는구나. 브랜치를 안전하게 다루고 병합할 수 있으니 우리 팀에 들어와서 바로 협업에 투입해도 코드 다 깨트릴 걱정은 없겠다!"

즉, 브랜치를 쪼개는 행위 자체가 "나는 실무 수준의 협업 프로세스를 몸으로 익힌 준비된 인재다"라는 것을 이력서 한 줄 없이 온몸으로 증명하는 최고의 기술 포트폴리오가 됩니다.

## 2. 실무 표준 브랜치 네이밍 규칙 🏷️
브랜치를 만들 때도 내 맘대로 `test`, `new-layout`, `modify-color`로 지으면 프로답지 못합니다. 실무에서는 브랜치 이름만 보고도 "**무슨 작업을 하고 있는지**" 직관적으로 알 수 있도록 규칙을 정해둡니다.

가장 널리 쓰이는 표준 접두사 규칙입니다.
- `feature/기능이름`: 새로운 기능 개발, UI 레이아웃 구현 등 (예: `feature/about`, `feature/projects`)
- `fix/수정이름`: 버그나 에러 수정 (예: `fix/mobile-menu`)
- `docs/문서이름`: 코드 수정이 아닌 리드미, 가이드 문서 수정 (예: `docs/readme-setup`)
- `refactor/개선이름`: 코드 리팩토링 (예: `refactor/clean-css`)
슬래시(`/`)를 사용하면 소스트리(Sourcetree)나 VS Code 같은 GUI 도구에서 브랜치들이 폴더 구조처럼 깔끔하게 묶여서 보이기 때문에 관리가 백 배는 쉬워집니다.

## 3. 실전! Feature Branch 워크플로우 🏃‍♂️
자, 우리의 포트폴리오 사이트의 첫 단추인 "기본 레이아웃 및 자기소개(About Me) 영역"을 개발해 봅시다. 이 과정을 현업 표준 브랜치 워크플로우에 맞춰서 진행해 보겠습니다.

#### 1단계: 언제나 출발지는 `main` 브랜치!
새로운 작업을 시작하기 전에는 항상 내비게이션 화살표(`HEAD`)가 가장 안전하고 최신 상태인 `main`에 머물고 있는지 확인해야 합니다.
```
# 1. 메인 브랜치로 이동합니다.
git switch main

# 2. 혹시 원격 저장소에 반영된 최신 코드가 있다면 받아옵니다.
git pull origin main
```

#### 2단계: 나만의 독립된 평행우주 생성하기
이제 자기소개 기능을 안전하게 개발할 feature/about 브랜치를 만들고 이동합니다.
```
# -c (create) 옵션을 주면 브랜치 생성과 동시에 이동까지 완료됩니다!
git switch -c feature/about
```

💡 **과거 명령어 주의**
- 예전에는 `git checkout -b feature/about`을 많이 썼습니다. 지금도 실무자들의 손에 익어 자주 쓰이지만, 최근 Git 버전에서는 기능이 명확히 분리된 `switch`를 권장합니다. 우리는 세련되게 `switch`를 씁니다!

#### 3단계: 안전한 우주에서 마음껏 코딩하기
이제 원본 코드가 망가질 걱정 없이 에디터에서 마음껏 코딩하세요!
프로젝트 루트 폴더에 `index.html` 파일을 만들고, 가볍게 자기소개 레이아웃을 작성해 봅니다.
```
<!-- index.html -->
<!DOCTYPE html>
<html lang="ko">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Awesome Portfolio</title>
</head>
<body>
    <header>
        <h1>안녕하세요, 신입 개발자 홍길동입니다 👋</h1>
        <p>기본기가 탄탄하고 시스템 안정성을 고민하는 엔지니어입니다.</p>
    </header>
</body>
</html>
```

#### 4단계: 작업 완료 후 커밋 남기기 (2편 컨벤션 적용!)
코드가 완성되었다면, 2편에서 배운 꼼꼼한 커밋 컨벤션을 적용하여 안전하게 기록을 저장합니다.
```
# 1. 상태를 먼저 확인해 봅니다. (빨간색으로 index.html이 뜰 것입니다.)
git status

# 2. 카트(Staging Area)에 담습니다.
git add index.html

# 3. 약속된 타입(feat)과 제목을 적어 커밋합니다.
git commit -m "feat: 기본 레이아웃 및 자기소개 영역 마크업 추가"
```

## 🎯 오늘 배운 내용 요약
1. 혼자 하는 프로젝트라도 **Feature Branch Flow**를 사용하면, 채용 담당자에게 실무 수준의 협업 능력을 증명하는 **아름다운 네트워크 그래프**를 선물할 수 있다.
2. 브랜치를 생성할 때는 슬래시(`/`)를 활용해 폴더 구조로 관리하자. (예: `feature/about`)
3. 브랜치를 새로 만들고 즉시 이동할 때는 `git switch -c 브랜치명`을 사용하는 것이 최근 트렌드이자 표준이다.

축하합니다! 원본을 안전하게 지켜가며 나만의 독립된 작업 우주에서 성공적으로 첫 기능을 개발했습니다.

그런데 이 개발된 코드를 어떻게 `main` 우주로 가져올까요? 그냥 내 컴퓨터에서 합쳐버리면 재미없겠죠?

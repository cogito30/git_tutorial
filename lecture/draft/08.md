$$실무 Git 튜토리얼 8편$$

코드 히스토리를 아름답게: Rebase 🛠️✨

안녕하세요! 지난 시간까지 우리는 Git의 필수 기능들을 익히며 훌륭한 주니어 개발자로 거듭나고 있었습니다.

하지만 실무에서 `git log`를 쳐보면, 여러분의 상상과는 다르게 커밋 히스토리가 거미줄처럼 복잡하게 얽혀 있는 것을 볼 수 있습니다. 여러 명이 동시에 `Merge`를 하다 보면 히스토리가 마치 복잡한 지하철 노선도처럼 변해버리죠.

이때 시니어 개발자분들이 종종 이런 말씀을 하십니다.
```
👨‍💻 사수: "A씨, 이번에 올리신 PR, 커밋이 너무 자잘하게 쪼개져 있네요. Rebase로 커밋 좀 하나로 묶어서(Squash) 다시 올려주실래요?"
```
순간 식은땀이 흐릅니다. `Rebase`? 그게 뭐죠?
오늘 우리는 Git 히스토리를 예술적으로 가꿔주는 마법의 명령어, **Rebase**에 대해 완벽하게 파헤쳐 보겠습니다.

## 1. Merge vs Rebase: "지하철 노선도냐, 일직선 고속도로냐" 🛣️

브랜치 두 개를 합치는 방법에는 Merge와 Rebase 두 가지가 있습니다.
- **Merge (병합)**: 두 갈래로 나뉘었던 브랜치를 하나로 모으면서 새로운 '병합 커밋(Merge Commit)'을 만듭니다. 히스토리(과거의 흔적)가 있는 그대로 100% 보존되지만, 선이 여러 갈래로 나뉘어 그래프가 지저분해집니다.
- **Rebase (재배치)**: 단어 뜻 그대로 'Base(시작점)를 다시(Re) 설정한다'는 뜻입니다. 내가 작업한 브랜치의 커밋들을 뚝 떼어다가, 원본 브랜치(`main`)의 가장 최신 끝단에 이어 붙입니다. 결과적으로 **한 줄로 깔끔하게 이어지는 일직선 히스토리**가 만들어집니다.

## 2. 기본 Rebase 사용해 보기 🧱
내가 `feature/login` 브랜치에서 작업하는 동안, 누군가 `main` 브랜치에 새로운 코드를 추가했다고 가정해 봅시다. 내 브랜치에 최신 `main` 코드를 깔끔하게 한 줄로 합쳐오고 싶다면 이렇게 합니다.
```
# 1. 내 작업 브랜치로 이동
git switch feature/login

# 2. main 브랜치를 최신 상태로 만들고 (fetch/pull 생략)
# main을 베이스로 삼아 내 커밋들을 그 위에 재배치!
git rebase main
```

#### 🚨 충돌(Conflict)이 났어요!
`Merge`할 때처럼 `Rebase` 중에도 충돌이 날 수 있습니다. 당황하지 말고 에디터에서 충돌을 해결한 뒤, `git add .`를 하고 `git rebase --continue`를 쳐주면 다음 커밋으로 넘어갑니다. (commit을 새로 하지 않는다는 점에 주의하세요!)

## 3. 🪄 실무 핵심
내 맘대로 커밋 주무르기: Interactive Rebase

사실 실무에서 `Rebase`를 쓰는 가장 큰 이유는 단순히 브랜치를 합치기 위해서가 아니라, 내 과거의 커밋들을 예쁘게 성형수술(`-i` 옵션)하기 위해서입니다.

개발을 하다 보면 이런 커밋들을 남길 때가 있습니다.
- `feat`: 로그인 기능 구현 중
- `fix`: 오타 수정
- `fix`: 진짜 오타 수정
- `feat`: 로그인 기능 완료

이 4개의 커밋을 그대로 PR에 올리면 리뷰어가 보기 피곤하겠죠? **이걸 하나의 깔끔한 커밋으로 뭉쳐보겠습니다.**

#### 🛠️ Step 1. Interactive 모드 실행
가장 최근 4개의 커밋을 조작하기 위해 아래 명령어를 칩니다. (`-i`는 interactive의 약자)
```
git rebase -i HEAD~4
```

#### 📝 Step 2. 에디터에서 커밋 조작하기 (Squash)
명령어를 치면 에디터(보통 vi 또는 VS Code)가 열리고, 과거 커밋 4개가 시간 역순(가장 오래된 것이 맨 위)으로 나타납니다.
```
pick 1a2b3c4 feat: 로그인 기능 구현 중
pick 5d6e7f8 fix: 오타 수정
pick 9g0h1i2 fix: 진짜 오타 수정
pick 3j4k5l6 feat: 로그인 기능 완료
```


앞에 적힌 `pick`(그대로 둔다)이라는 글자를 지우고, 합치고 싶은 커밋 앞에 `squash`(또는 `s`)라고 적어줍니다. `squash`는 "내 커밋을 바로 위(이전) 커밋에 뭉뚱그려 합치겠다!"라는 뜻입니다.
```
pick 1a2b3c4 feat: 로그인 기능 구현 중
s 5d6e7f8 fix: 오타 수정
s 9g0h1i2 fix: 진짜 오타 수정
s 3j4k5l6 feat: 로그인 기능 완료
```

#### ✨ Step 3. 새로운 커밋 메시지 작성
저장하고 창을 닫으면, 다시 새로운 창이 열리며 4개가 합쳐진 새로운 커밋 메시지를 적으라고 합니다. 기존 내용을 다 지우고 깔끔하게 하나만 적습니다.
`feat: 카카오 소셜 로그인 기능 완벽 구현`
저장하고 닫으면 끝입니다! `git log`를 쳐보세요. 지저분했던 4개의 커밋이 사라지고 영롱하고 깔끔한 커밋 1개만 남아있을 것입니다. PR을 올릴 준비가 완벽하게 끝났습니다!

## 4. 💀 경고
Rebase의 절대 황금률 (이것만은 제발!)

Rebase는 히스토리를 조작하는 강력한 마법이지만, 그만큼 치명적인 부작용이 있습니다. 실무에서 이 규칙을 어기면 팀 전체의 코드가 꼬여버리는 대참사가 일어납니다.

**🚫 "이미 GitHub(원격 저장소)에 Push해서 남들과 공유된 커밋은 절대 Rebase 하지 마라!"**

Rebase는 겉보기엔 같아 보여도 내부적으로는 '완전히 새로운 커밋(주민등록번호가 다름)'을 만들어냅니다.
만약 다른 팀원들이 이미 내 코드를 다운받아 갔는데 내가 몰래 Rebase로 히스토리를 바꿔버리면, 팀원들의 Git은 "어? 내가 알던 과거랑 다른데?" 하며 엄청난 충돌과 혼란을 뿜어냅니다.

#### 💡 실무 안전 가이드
Rebase(특히 `squash` 등)는 **오직 내 컴퓨터(Local)에만 있는, 아직 아무에게도 공유하지 않은 나만의 브랜치**에서 PR을 올리기 직전, 코드를 예쁘게 다듬을 때만 사용하세요! (만약 내 원격 브랜치에 혼자 올려둔 걸 수정했다면 `git push -f` 로 강제 푸시를 해야 합니다.)

## 🎯 오늘 배운 내용 요약
1. Merge는 히스토리를 있는 그대로 보존하고, Rebase는 히스토리를 일직선으로 깔끔하게 만든다.
2. `git rebase -i HEAD~숫자`를 통해 과거의 커밋 여러 개를 하나로 뭉치거나(Squash), 메시지를 수정할 수 있다.
3. **절대 주의**: 이미 협업 중인 원격 저장소에 올라간 코드는 절대 Rebase 하면 안 된다!

이제 여러분은 남들에게 보여주기 부끄러운 커밋들을 몰래(?) 예쁘게 포장할 수 있는 능력을 갖췄습니다. 리뷰어들에게 칭찬받는 깔끔한 PR을 만들어보세요!

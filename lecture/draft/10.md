$$실무 Git 튜토리얼 10편$$

"방금 커밋 잘못했어요!": 취소의 미학 ⏪

안녕하세요! 지난 9편까지 우리는 브랜치를 다루고 코드를 합치는 고급 기술들을 마스터했습니다.

하지만 아무리 Git을 잘 다루는 시니어 개발자라도 실수는 하는 법입니다. 엔터키를 세게 치며 커밋을 날렸는데, 1초 뒤에 치명적인 오타나 빠뜨린 파일을 발견하면 눈앞이 캄캄해지죠.

"어떡하지? 그냥 새로 커밋 하나 더 만들까? '진짜 최종 오타 수정'이라고 올리면 사수한테 혼날 텐데..."

걱정하지 마세요! Git에는 우리의 흑역사를 지워줄 훌륭한 '되돌리기' 기능들이 있습니다. 오늘은 여러분이 마주할 3가지 실수 상황과 그에 맞는 처방전을 알려드립니다.

💊 상황 1. "방금 한 커밋 메시지에 오타가 났어요!"

📝 나의 실수: git commit -m "feat: 로긘 기능 추가" (아차, 로그인인데!)

가장 흔하게 발생하는 상황입니다. 코드는 완벽한데 커밋 메시지에 오타가 났거나, 방금 커밋에 포함시켰어야 할 파일 하나를 빼먹고 add를 안 했을 때 쓰는 마법의 명령어는 --amend (수정하다) 입니다.

🪄 처방전: git commit --amend

# 1. 메시지만 쏙 바꾸고 싶을 때
git commit --amend -m "feat: 로그인 기능 추가"



만약 파일을 빼먹고 커밋했다면 이렇게 하시면 됩니다.

# 1. 빼먹은 파일을 뒤늦게 카트에 담습니다.
git add 빠뜨린파일.js

# 2. 방금 전 커밋에 슬쩍 끼워 넣습니다. (메시지 그대로 유지)
git commit --amend --no-edit



이렇게 하면 새로운 커밋이 지저분하게 추가되지 않고, 방금 남겼던 마지막 커밋 하나가 깔끔하게 수정됩니다.

💊 상황 2. "아예 방금 한 커밋을 취소하고 과거로 갈래!"

📝 나의 실수: 앗, 실수로 아직 완성도 안 된 코드를 커밋해 버렸어! 커밋 자체를 없던 일로 하고 싶어!

마지막 커밋을 포함해 특정 시점으로 아예 타임머신을 타고 돌아가고 싶을 때는 reset을 사용합니다. reset은 과거로 돌아가면서 그 이후의 커밋 기록(영수증)을 아예 찢어버립니다.

과거로 돌아갈 때, '내가 짜던 코드'를 어떻게 할 것인지에 따라 3가지 옵션이 있습니다.

🪄 처방전: git reset (3가지 옵션)

돌아갈 목적지(과거 커밋 번호)를 지정하거나, HEAD~1 (현재 위치에서 1개 이전 커밋으로)을 사용합니다.

1. --soft : "영수증만 버리고, 카트(Staging)에 담아둔 채로 줘"

git reset --soft HEAD~1



커밋은 취소되지만, 내가 짠 코드는 그대로 유지되고 초록색(add된 상태)으로 카트에 예쁘게 담겨 있습니다. 다시 묶어서 커밋하기 가장 좋은 상태입니다.

2. --mixed (기본값) : "카트에서도 빼고, 작업 공간(Working)에 던져놔"

git reset HEAD~1  # 또는 git reset --mixed HEAD~1



커밋도 취소되고 카트에서도 빠집니다. 코드는 빨간색(add 안 된 상태)으로 내 에디터에 그대로 남아있습니다. 코드를 꽤 많이 수정하고 다시 카트에 담아야 할 때 씁니다.

3. --hard : "다 꼴 보기 싫어! 내가 짠 코드까지 싹 다 날려버려!!" 💀

git reset --hard HEAD~1



$$🚨 실무 주의사항$$

 커밋 기록은 물론이고, 방금 전까지 내가 에디터에서 열심히 쳤던 코드들까지 영구적으로 날아갑니다. 정말 코드를 처음부터 다시 짜고 싶을 때만 신중하게 쓰세요!

💊 상황 3. "이미 원격 저장소(GitHub)에 Push를 해버렸는데요...?"

📝 나의 실수: main 브랜치에 코드를 밀어 올렸는데 서비스에 심각한 버그가 터졌습니다. 코드를 어제 상태로 당장 되돌려야 해요!

여기서 아주 중요한 실무 황금률이 다시 등장합니다. (8편 Rebase에서 배웠던 것과 같습니다.)
이미 GitHub에 올라가서 남들과 공유된 커밋은 절대 reset으로 기록을 지워버려서는 안 됩니다! 남들의 Git이 고장 나버리기 때문이죠.

이때는 기록을 찢어버리는 대신, "이전 커밋의 내용을 정반대로 취소하는 새로운 커밋"을 추가하는 방식을 써야 합니다. 이것이 바로 revert 입니다.

🪄 처방전: git revert

# 1. 취소하고 싶은 '버그 커밋'의 번호(Hash)를 확인합니다.
git log

# 2. 해당 커밋을 취소(revert)하는 새로운 커밋을 만듭니다.
git revert 버그커밋번호



명령어를 치면 에디터가 열리면서 Revert "feat: 로그인 기능 추가" 같은 새로운 커밋 메시지가 자동으로 작성됩니다. 그대로 저장하고 나오면 됩니다.

결과적으로 코드는 과거 상태로 안전하게 돌아갔지만, git log를 보면 기록을 덮어 가린 것이 아니라 "A기능 추가함 -> A기능 다시 취소함" 이라는 모든 히스토리가 정직하게 남게 됩니다. 실무에서 이미 배포된 코드를 롤백(Rollback)할 때 가장 많이 쓰는 안전한 방법입니다!

🎯 오늘 배운 내용 요약 (멘붕 탈출 가이드)

amend: 방금 한 커밋의 메시지나 파일을 살짝 수정하고 싶을 때 (Push 전)

reset: 내가 방금 한 커밋을 아예 없던 일로 하고 과거로 갈 때 (Push 전)

revert: 이미 남들과 공유한(Push) 커밋을 안전하게 취소하고 싶을 때 (Push 후)

이제 여러분은 어떤 실수를 하더라도 침착하게 대응할 수 있는 Git 응급 처치 기술을 가지게 되었습니다.

하지만... 만약 술을 마시고 코딩하다가 git reset --hard를 쳐버려서 3일 치 코드가 싹 날아갔다면 어떡하죠? 😱

$$실무 Git 튜토리얼 3편$$

브랜치(Branch): 내 작업 공간 분리하기 🌿

안녕하세요! 지난 2편에서는 코드를 수정하고 찰칵! 사진을 찍어 저장하는 기본 사이클(`add`, `commit`)을 배웠습니다.

그런데 실무에서는 이런 일이 자주 발생합니다.
```
👨‍💻 사수: "A씨, 지금 서비스되고 있는 메인 코드 건드리지 말고, 새 폴더 복사해서 거기서 '결제 기능' 좀 만들어주세요."
🧑‍💻 나: (폴더를 통째로 복사하며) 프로젝트_결제기능_진짜최종_복사본...
```
Git을 사용하면 이렇게 무식하게 폴더를 복사할 필요가 없습니다! 바로 오늘 배울 마법 같은 기능, '브랜치(Branch)'가 있기 때문이죠.

## 1. 브랜치(Branch)란? "코딩 평행우주 만들기" 🌌

브랜치(Branch)는 영어로 '나뭇가지'라는 뜻입니다. 원본 코드(보통 `main` 브랜치)라는 굵은 기둥에서 나뭇가지처럼 뻗어 나와, **나만의 독립적인 작업 공간**을 만드는 기능입니다.

어벤져스 같은 영화에 나오는 '평행우주'를 떠올려보세요.
여러분은 현재 안전한 `main` 우주에 있습니다. 여기서 새로운 기능을 만들고 싶은데 코드를 망칠까 봐 겁이 납니다. 이때 `feature`라는 새로운 우주를 하나 복제해서 만들어 냅니다. 이 새로운 우주에서는 코드를 마음껏 지지고 볶고 폭파시켜도, 원래 있던 `main` 우주의 코드에는 **1%의 영향도 주지 않습니다**. 완벽하게 분리되어 있으니까요!

#### 💡 잠깐, HEAD가 뭔가요?

Git 명령어 결과를 보다 보면 `HEAD`라는 단어를 종종 보게 됩니다. 어렵게 생각할 것 없이 "현재 내가 머물고 있는 우주(브랜치)를 가리키는 내비게이션 화살표(📍)"라고 생각하시면 됩니다.

## 2. 브랜치 실전 명령어: 만들고, 이동하고, 지우기
자, 그럼 평행우주를 직접 만들고 넘나들어 볼까요?

#### 🌿 Step 1. 내 브랜치 확인하기 & 만들기: `git branch`
현재 프로젝트에 어떤 브랜치들이 있는지 확인합니다.
```
git branch
```
처음이라면 `* main` 하나만 초록색으로 뜰 겁니다. 앞의 `*` 기호가 바로 내비게이션(`HEAD`)입니다. "너 지금 main 브랜치에 있어!"라는 뜻이죠.

이제 새로운 결제 기능을 만들기 위해 `pay`라는 브랜치를 만들어보겠습니다.
```
git branch pay
```
다시 `git branch`를 쳐보면 `main`과 `pay` 두 개가 보일 겁니다. 하지만 별표(`*`)는 여전히 `main`에 있습니다. 우주를 만들기만 하고 아직 이동하진 않은 상태입니다.

#### 🚀 Step 2. 브랜치 이동하기: git switch
방금 만든 `pay` 브랜치로 내비게이션(`HEAD`)을 이동시켜 봅시다.
```
git switch pay
```
이제 코드를 수정하고 `add`, `commit`을 하면 그 기록은 오직 `pay` 브랜치에만 남습니다!

#### 🚨 잠깐! `git checkout`은 뭔가요?
- 과거에는 브랜치를 이동할 때 `git checkout pay`를 썼습니다. 하지만 `checkout`이 너무 많은 기능을 가지고 있어서 초보자들이 헷갈려하자, 최근 Git 버전에서는 브랜치 이동 전용 명령어인 `git switch`가 새로 탄생했습니다. 실무에 계신 시니어분들은 손에 익은 `checkout`을 여전히 많이 쓰지만, 여러분은 직관적인 `switch`를 쓰는 것을 추천합니다!

$$⭐ 실무 꿀팁$$

만들고 바로 이동하기 (가장 많이 씀!)

실무에서는 브랜치를 만들고 이동하는 것을 따로 치기 귀찮아서, 옵션을 달아 한 방에 처리합니다. **이 명령어를 제일 많이 씁니다.**
```
git switch -c pay  # (checkout -b pay 와 같습니다)
```
`-c`는 create의 약자입니다. "pay 브랜치를 만들고(create) 바로 거기로 이동해!"라는 뜻이죠.

#### 🗑️ Step 3. 다 쓴 브랜치 삭제하기: `git branch -d`

`pay` 브랜치에서 작업을 다 끝내고 코드를 `main`에 무사히 합쳤다면(합치는 법은 다음 시간에 배웁니다!), 이제 필요 없는 평행우주는 삭제해서 깔끔하게 유지하는 것이 좋습니다.
```
# 먼저 삭제할 브랜치 밖으로 나와야 합니다. (main으로 이동)
git switch main

# pay 브랜치 삭제
git branch -d pay
```

$$💡 실무 꿀팁$$

"아무 이름이나 쓰지 마세요!" 브랜치 네이밍 룰

지난 시간에 커밋 메시지에 말머리(`feat:`, `fix:`)를 달았던 것 기억하시나요? 브랜치를 만들 때도 내 맘대로 `test`, `new_branch` 라고 지으면 사수에게 불려 갑니다. 😅

실무에서는 다른 팀원이 브랜치 이름만 봐도 **"아, 이 사람이 지금 무슨 작업을 하고 있구나"** 알 수 있도록 룰을 정해둡니다. 슬래시(`/`)를 사용해 폴더처럼 구조화하는 것이 일반적입니다.

- `feature/기능이름` : 새로운 기능 개발 (예: `feature/login`, `feature/payment`)
- `bugfix/이름` : 일반적인 버그 수정 (예: `bugfix/login-error`)
- `hotfix/이름` : 라이브 서버의 긴급 치명적 버그 수정 (예: `hotfix/server-crash`)
- `release/버전` : 다음 버전을 배포하기 위한 준비 (예: `release/v1.2.0`)

```
👨‍💻 좋은 예시: `git switch -c feature/shopping-cart`
❌ 나쁜 예시: `git switch -c my-work-2`
```

## 🎯 오늘 배운 내용 요약
1. 브랜치는 원본 코드를 망치지 않고 작업할 수 있는 **평행우주**다.
2. 브랜치 만들고 바로 이동하기 👉 `git switch -c 브랜치명`
3. 브랜치 이름도 규칙(예: `feature/로그인`)을 지켜서 깔끔하게!

자, 이제 나만의 공간에서 마음껏 코딩할 수 있게 되었습니다!
그런데 `feature/login` 우주에서 결제 기능을 다 만들었다면, **이 코드를 다시 원래 세계인 `main` 우주로 가져와서 합쳐야겠죠?**

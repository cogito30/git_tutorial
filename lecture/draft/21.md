나 자신과 협업하기: PR과 셀프 코드 리뷰 (Self-Review) 🔍

안녕하세요! 지난 3편에서 우리는 원본 브랜치를 안전하게 지키면서 독립된 평행우주인 `feature/about` 브랜치를 만들고 첫 번째 UI 레이아웃을 성공적으로 개발해 보았습니다.

이제 이 코드를 실제 서비스가 돌아갈 메인 우주인 `main` 브랜치로 가져와서 합쳐야 합니다.

설마 아직도 내 컴퓨터 터미널에서 `git checkout main`을 친 뒤, `git merge feature/about`으로 슥 합쳐버리고 끝내시나요? **실무에서는 절대로 로컬(내 컴퓨터)에서 임의로 브랜치를 합치지 않습니다**.

오늘은 1인 프로젝트의 퀄리티를 대기업 협업 수준으로 끌어올리는 비밀 병기, **Pull Request(PR)와 셀프 코드 리뷰(Self-Code Review)** 문화를 우리 포트폴리오에 이식해 보겠습니다.

## 1. "혼자 작업하는데 PR을 왜 올리나요?" 🤔
가장 설득력 있는 질문입니다. "내가 짜서 내가 합치는데, 굳이 GitHub 웹사이트에 들어가서 게시글(PR)을 쓰고 혼자 버튼을 누르는 게 무슨 의미가 있죠?"

여기에는 채용 담당자(리드 개발자)들의 심금을 울리는 **3가지 엄청난 비밀**이 숨겨져 있습니다.

1. **지식의 기록 (개발 이력서 자동화)**: 시간이 지나면 내가 이 코드를 왜 이렇게 짰는지 까먹기 마련입니다. PR은 "내가 이 기능을 왜 만들었고, 어떤 고민을 거쳐 어떻게 구현했는지"를 남기는 최고의 개발 일지(Documentation)가 됩니다.

2. **협업 DNA 증명**: 리드 개발자가 여러분의 GitHub 저장소에 들어와 가장 먼저 누르는 탭 중 하나가 바로 **\[Pull Requests\] -> \[Closed\]** 입니다. 여기에 정돈된 PR 기록이 쌓여 있다면, 담당자는 코드를 보기도 전에 이렇게 판단합니다.
- "이 지원자는 PR 템플릿을 쓸 줄 알고, 리뷰어 배려 능력이 있구나. 우리 팀에 들어와서 바로 협업 프로세스에 녹아들 수 있겠다."

3. **제3의 눈으로 코드 바라보기 (버그 원천 차단)**: 에디터에서 코딩할 때는 안 보이던 스파게티 코드, 지저분한 콘솔 로그, 오타들이 **GitHub의 \[Files changed\] 화면**으로 보면 마법처럼 눈에 아주 잘 들어옵니다. 내 코드를 객관적으로 검증할 수 있는 마지막 안전장치가 바로 PR 단계입니다.

## 2. 🔄 실무 표준 PR 워크플로우 5단계
자, 그럼 우리가 지난번에 작업한 `feature/about` 브랜치의 결과물을 가지고 실무 표준 PR 과정을 그대로 밟아봅시다.

#### 1단계: 원격 저장소로 내 우주 쏘아 올리기 (Push)
로컬에만 있는 평행우주 브랜치를 GitHub 서버로 밀어 올립니다.
```
# 내 로컬의 feature/about 브랜치를 원격(origin)으로 push 합니다.
git push origin feature/about
```

#### 2단계: GitHub에서 [Compare & pull request] 클릭하기
푸시를 완료하고 내 GitHub 저장소 페이지에 접속하면, 상단에 노란색 알림 띠와 함께 [Compare & pull request]라는 영롱한 버튼이 나타납니다. 이 버튼을 클릭합니다.

#### 3단계: '사랑받는 주니어'의 PR 템플릿 작성하기
제목만 대충 쓰고 올리는 것은 금물입니다. 리뷰어(혹은 훗날의 나 자신)가 한눈에 변경 사항을 파악할 수 있도록 구조화된 설명서를 작성해야 합니다.

아래 템플릿을 복사해서 PR 본문(Description)에 붙여넣고 작성해 보세요.
```
## 📝 작업 내용 (What)
- 메인 화면 자기소개 영역(About Me) 마크업 구현
- 반응형 웹 디자인 적용 (모바일, 데스크톱 레이아웃 분리)
- GitHub, 블로그 등으로 이동하는 SNS 아이콘 링크 배치

## 💡 고민한 점 & 해결 과정 (Why & How)
- **고민:** 모바일 화면에서 프로필 사진과 글씨가 겹치는 현상이 발생했습니다.
- **해결:** Tailwind CSS의 브레이크포인트(`md:flex-row`)를 활용하여 모바일에서는 세로로, 태블릿 이상에서는 가로로 레이아웃이 자연스럽게 전환되도록 수정했습니다.

## 📸 스크린샷 (선택)
| 모바일 뷰 | 데스크톱 뷰 |
| :---: | :---: |
| 여기에 모바일 캡처 이미지 드래그 | 여기에 데스크톱 캡처 이미지 드래그 |

## 🔍 리뷰어에게 당부하고 싶은 말
- `index.html` 45번째 줄에 사용한 Flexbox 정렬 방식이 웹 표준에 맞게 깔끔하게 짜였는지 검토해 주시면 감사하겠습니다!
```

이렇게 정성스러운 PR을 올리는 훈련을 반복하면, 실제 취업 후 첫 코드 리뷰를 받을 때 엄청난 칭찬을 듣게 될 것입니다.

## 3. 🛠️ 셀프 코드 리뷰 (Self-Code Review)의 정석
PR을 생성했다면 바로 머지(Merge) 버튼을 누르고 싶겠지만, 잠시 심호흡을 하세요. 이제 여러분은 '**개발자'에서 '엄격한 시니어 리뷰어'로 빙의**할 시간입니다.

PR 페이지 상단의 **\[Files changed\]** 탭을 클릭합니다.

#### 📋 셀프 리뷰 체크리스트 5가지
- \[ \] 불필요한 디버깅 코드 제거: 테스트용으로 쓰고 지우지 않은 console.log()나 color: red 같은 임시 스타일이 남아있지 않은가?
- \[ \] 일관된 들여쓰기 및 포맷팅: 태그가 안 닫혔거나, 들여쓰기가 뒤죽박죽인 곳은 없는가?
- \[ \] 의미 있는 네이밍: 변수명이나 클래스명이 역할을 직관적으로 설명하고 있는가? (temp-box 대신 profile-container 등)
- \[ \] 시맨틱 마크업: 그냥 div만 남발하지 않고 header, section, main, footer 같은 의미 있는 HTML 태그를 적절히 사용했는가?

#### 💬 내 코드에 직접 댓글 남기기 (신의 한 수)

셀프 리뷰를 하다가 스스로 아쉽거나 더 고민해 보고 싶은 라인이 있다면, **그 줄 왼쪽의 \[+\] 버튼을 눌러 직접 댓글**을 남겨보세요.

🙋‍♂️ **나의 댓글**
- "현재는 이미지 파일 경로를 로컬로 지정해 두었는데, 나중에 AWS S3나 CDN 주소로 마이그레이션할 때 이 부분을 리팩토링할 예정입니다."

채용 담당자가 이 이력을 보게 된다면 머릿속으로 무릎을 탁 치며 감탄할 것입니다. **"이 지원자는 자기 코드의 한계점을 명확히 인지하고 있고, 개선 방향까지 설계할 줄 아는 사람이구나!"** 라는 깊은 인상을 남길 수 있죠.

## 4. 🎉 무사히 합치기 (Merge)
스스로의 검증을 통과했다면 이제 드디어 코드를 main 브랜치에 합칠 시간입니다.

PR 페이지 하단의 초록색 **\[Merge pull request\]** 버튼을 클릭하고 **\[Confirm merge\]**를 누릅니다. 코드가 성공적으로 병합되면 로컬에서 길을 잃지 않도록 사용이 끝난 원격 브랜치를 삭제해 줍니다. (**\[Delete branch\]** 버튼 클릭)

원격 저장소의 `main`에 코드가 성공적으로 합쳐졌다면, 내 컴퓨터에서도 최신 상태를 동기화해 줍니다.
```
# 로컬의 main 브랜치로 이동합니다.
git switch main

# GitHub 서버에서 성공적으로 병합된 최신 코드를 다운받아 적용합니다.
git pull origin main

# 사용이 끝난 로컬의 feature/about 브랜치도 깔끔하게 지워줍니다.
git branch -d feature/about
```

## 🎯 오늘 배운 내용 요약

1. 혼자 하는 프로젝트라도 **Pull Request(PR)는 나의 개발 역량을 정교하게 기록하는 최고의 도구**다.
2. PR 본문에는 작업 내용(What), 고민 사항(Why & How), 이미지(Screenshots)를 꼼꼼하게 채워 넣어 협업 능력을 증명하자.
3. 머지하기 전, **\[Files changed\] 탭에서 셀프 코드 리뷰**를 진행하며 남겨둔 지저분한 코드를 정리하고 스스로 피드백 댓글을 달아 전문성을 어필하자.
